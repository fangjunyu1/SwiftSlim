# Запазване на сумата в касичката

В предишния урок реализирахме проста „касичка“. В този урок ще научим как да направим така, че сумата в нея да се запазва постоянно.

Този урок е посветен основно на `UserDefaults` и property wrapper-а `@AppStorage`, като също така ще разширим темата с ролята на `onAppear` и употребата на Optional типове (`nil`).

Важно е да отбележим, че този урок продължава върху кода на „касичката“, разработен в предишния урок.

## Проблемът с постоянното запазване

В момента интерфейсът на „касичката“ е много изчистен и удобен за използване.

![Piggy Bank](../../Resource/014_view.png)

Но има един очевиден проблем: всеки път, когато обновим изгледа или затворим приложението, общата сума в „касичката“ се връща на `0`, а всички предишни спестявания изчезват.

Това означава, че данните всъщност не се запазват наистина. С други думи, настоящата „касичка“ може да пази само временни данни.

### Защо не може да се запазва постоянно?

Това е така, защото използваме променлива, декларирана с `@State`:

```swift
@State private var amount = 0
```

Променливата, декларирана с `@State`, зависи изцяло от жизнения цикъл на изгледа.

Когато изгледът бъде създаден, `amount` се инициализира със `0`, а когато изгледът бъде унищожен, `amount` също се унищожава.

Данните в `amount` съществуват само в паметта и не се запазват на устройството.

Ако не искаме `amount` да бъде обвързан с жизнения цикъл на изгледа, трябва да реализираме постоянно съхранение на данните. Или с други думи, да ги запазим на устройството.

### Какво е „постоянно съхранение на данни“

Постоянното съхранение на данни може да се разбере като преместване на данни от „временната памет“ към „паметта на устройството“.

Когато затворим изгледа или приложението, данните няма да бъдат загубени.

В Swift, ако искаме да реализираме просто постоянно съхранение, можем да използваме `UserDefaults`, а за по-сложни данни трябва да използваме `SwiftData` или `CoreData`.

В този урок първо ще научим най-простото решение, а именно `UserDefaults`.

## UserDefaults

`UserDefaults` се използва за запазване на леки данни под формата на двойки ключ-стойност и често се използва за типове като `String`, `Int`, `Double`, `Bool` и други основни типове.

### Запазване на данни

`UserDefaults` използва метода `set`, за да запазва данни:

```swift
UserDefaults.standard.set("FangJunyu", forKey: "name")
UserDefaults.standard.set(28, forKey: "age")
UserDefaults.standard.set(true, forKey: "isMember")
```

Първият параметър на `UserDefaults` е самата стойност, която искаме да запазим, тук например `String`, `Int` и други типове.

Вторият параметър `forKey` е подобен на име на променлива и служи за идентифициране на съхранените данни.

### Четене на данни

`UserDefaults` използва съответни методи за четене:

```swift
let name = UserDefaults.standard.string(forKey: "name")  // FangJunyu
let age = UserDefaults.standard.integer(forKey: "age")   // 28
let isMember = UserDefaults.standard.bool(forKey: "isMember") // true
```

При четене трябва задължително да използваме подходящия метод за съответния тип, иначе е възможно да получим грешка или неточни данни.

### Важни неща за UserDefaults

#### 1. Връщане на Optional тип

Когато четем данни чрез `UserDefaults`, някои методи връщат Optional тип.

Например:

```swift
let name = UserDefaults.standard.string(forKey: "name")
```

Типът на `name` е:

```swift
String?
```

Това означава, че може да има стойност, но може и да няма.

**Защо връща Optional тип?**

Защото даден ключ в `UserDefaults` може изобщо да няма запазени данни.

Например:

```swift
let name = UserDefaults.standard.string(forKey: "City") // nil
```

Ако никога преди това не сме запазвали данни с ключ `"City"`, резултатът ще бъде:

```swift
nil
```

Важно е да подчертаем, че `nil` означава липса на данни, а не празен низ.

В урока за „системата на типовете“ вече срещнахме подобен проблем: когато преобразуването на тип не успее, се връща Optional.

В такъв случай трябва да използваме `??`, за да дадем стойност по подразбиране и да разопаковаме стойността:

```swift
let name = UserDefaults.standard.string(forKey: "City") ?? "Rizhao"
```

Ако `UserDefaults` прочете стойност, ще използваме нея; ако не може да прочете нищо (`nil`), ще използваме стойността по подразбиране.

#### 2. Връщане на не-Optional тип

При четене на `Int`, `Double` и `Bool`, `UserDefaults` връща не-Optional тип.

Например:

```swift
let num = UserDefaults.standard.integer(forKey: "num")   // 0
let height = UserDefaults.standard.double(forKey: "height") // 0.0
let isBuyCar = UserDefaults.standard.bool(forKey: "isBuyCar") // false
```

Дори ако за тези ключове никога не е имало запазени данни, върнатата стойност няма да е `nil`.

Вместо това ще бъде върната стойност по подразбиране:

```text
Int → 0
Double → 0.0
Bool → false
```

#### 3. Ограничение на количеството данни

`UserDefaults` е подходящ само за малки обеми от данни, като настройки на приложението или прости състояния.

В реални приложения той не е подходящ за големи количества данни или за данни, които се четат и записват много често.

## Запазване на сумата в „касичката“

Сега можем да използваме `UserDefaults`, за да реализираме логиката за запазване на общата сума в „касичката“.

Когато потребителят натисне бутона, въведената сума ще се добави към общата, а след това общата сума ще бъде запазена в `UserDefaults`.

```swift
Button("Save") {
    amount += number
    number = 0
    UserDefaults.standard.set(amount, forKey: "amount")    // UserDefaults
}
.buttonStyle(.borderedProminent)
```

С това вече реализирахме логиката за запис на данните.

## Обяснение на логиката за постоянно съхранение

Макар че вече сме реализирали записването на данните, при реално стартиране ще забележим, че всеки път при обновяване на изгледа или повторно отваряне на приложението променливата `amount` отново е `0`.

Причината е:

```swift
@State private var amount = 0
```

В SwiftUI променливите с `@State` се инициализират наново при създаване на изгледа.

Макар стойността на `amount` да е запазена в `UserDefaults`, когато изгледът се зареди наново, `amount` пак ще бъде върнат на `0`.

Това означава, че в момента само записваме данните в `UserDefaults`, но не ги възстановяваме обратно в изгледа.

Затова, когато изгледът се зарежда, трябва ръчно да прочетем данните от `UserDefaults` и отново да ги присвоим на `amount`, за да завършим цялата логика на постоянното съхранение.

### Логическа аналогия

Можем да разберем този процес чрез аналогия с „дъската в класната стая“:

По време на урок учителят пише съдържанието на дъската. Това е подобно на текущото състояние в изгледа (`@State`).

Когато урокът свърши, дъската се изтрива, за да се подготви за следващия час. Това е еквивалентно на унищожаването на изгледа, при което и данните в `@State` се изчистват.

За да не бъде изгубено учебното съдържание, учителят го запазва предварително в материалите за урока. Тези материали отговарят на данните, запазени в `UserDefaults`.

Когато следващият час започне отново, дъската е празна (`@State` след инициализация), и учителят трябва отново да напише предишното съдържание на дъската според запазения материал (тоест чрез четене от `UserDefaults`).

Важно е да разберем, че дъската не възстановява старото съдържание автоматично. То трябва да бъде написано отново чрез „прочитане на учебния материал“ (тоест чрез четене от `UserDefaults`).

## Използване на `onAppear` за четене на данните

Когато изгледът се покаже, трябва ръчно да прочетем стойността от `UserDefaults` и да я присвоим на `amount`, деклариран с `@State`, за да реализираме истинско постоянно съхранение.

В SwiftUI можем да използваме `onAppear`, за да изпълним инициализираща логика при показване на изгледа:

```swift
.onAppear {}
```

Поставяме кода за четене на `UserDefaults` в `onAppear`:

```swift
.onAppear {
    amount = UserDefaults.standard.integer(forKey: "amount")
}
```

Когато изгледът се създаде, `amount` първоначално е `0`; когато изгледът се покаже, `onAppear` се задейства, прочита стойността от `UserDefaults` и я присвоява на `amount`. Така изгледът може да покаже последната запазена обща сума.

С това цялата логика за постоянно съхранение в „касичката“ вече е завършена.

## Пълен код

```swift
struct ContentView: View {
    
    @State private var amount = 0
    @State private var number = 0
    
    var body: some View {
        VStack {
            Text("\(amount)")
                .font(.largeTitle)
                .fontWeight(.bold)
            TextField("Amount", value: $number, format: .number)
                .frame(width: 100)
                .padding(5)
                .border(Color.black, width: 1)
            Button("Save") {
                amount += number
                number = 0
                UserDefaults.standard.set(amount, forKey: "amount")    // UserDefaults
            }
            .buttonStyle(.borderedProminent)
        }
        .background {
            Image("1")
        }
        .onAppear {
            amount = UserDefaults.standard.integer(forKey: "amount")
        }
    }
}
```

## Property wrapper-ът `@AppStorage`

SwiftUI предоставя property wrapper-а `@AppStorage`, който може да опрости логиката за запазване чрез `UserDefaults`.

```swift
@AppStorage("amount") private var amount = 0
```

`@AppStorage` е обвивка върху `UserDefaults`, а `"amount"` е съответният ключ в `UserDefaults`.

Когато изгледът се създаде, данните автоматично се четат от `UserDefaults`, а когато стойността на променливата се промени, те автоматично се записват обратно.

Той също така поддържа механизма за обновяване на състоянието в SwiftUI и подобно на `@State`, когато стойността се промени, SwiftUI изгледът ще се обнови.

С други думи, `@AppStorage` може сам да реализира логиката за четене и запис на данните, без да се налага ръчно да извикваме `UserDefaults` и без да използваме `onAppear`.

Затова можем да заменим кода с `UserDefaults` по следния начин:

```swift
struct ContentView: View {
    
    @AppStorage("amount") private var amount = 0
    @State private var number = 0
    
    var body: some View {
        VStack {
            Text("\(amount)")
                .font(.largeTitle)
                .fontWeight(.bold)
            TextField("Amount", value: $number, format: .number)
                .frame(width: 100)
                .padding(5)
                .border(Color.black, width: 1)
            Button("Save") {
                amount += number
                number = 0
            }
            .buttonStyle(.borderedProminent)
        }
        .background {
            Image("1")
        }
    }
}
```

## Обобщение

В този урок научихме основно следните неща:

Първо, `UserDefaults` се използва за запазване на леки данни и ние го използвахме, за да запазим общата сума на устройството. `UserDefaults` предлага методи за четене и запис и изисква да умеем да работим и с Optional типове в определени случаи.

Второ, променливите с `@State` се използват само за временно състояние вътре в изгледа и техният жизнен цикъл е свързан с този на самия изглед. Когато изгледът се създаде отново, `@State` също се инициализира отново.

При възстановяването на данни от `UserDefaults` научихме как да използваме `onAppear`, за да прочетем данните при показване на изгледа и така да реализираме възстановяването.

Накрая използвахме `@AppStorage`, за да заменим ръчната логика с `UserDefaults`, така че данните да се четат автоматично при зареждане на изгледа и да се записват автоматично при промяна на стойността. Така отпада нуждата от ръчно извикване на `UserDefaults` и от логиката в `onAppear`.

С тези знания можем да реализираме базова функция за съхранение на данни в приложението, така че данните да бъдат запазвани постоянно.

## Упражнения след урока

Сега „касичката“ поддържа запазване на сумата, но функционалността все още не е пълна.

1. Ако потребителят иска да изтегли сума, трябва да проектираме съответната логика.
2. Ако потребителят иска да изчисти цялата сума, трябва да реализираме логика за нулиране.
3. Ако искаме да добавим име на касичката, трябва да включим и запазване на данни от тип `String`.

Можеш да надградиш тези функции върху текущата основа, за да направиш „касичката“ по-завършена.

## Допълнителни знания - проблемът с началната стойност в полето за въвеждане

Сега можем допълнително да подобрим взаимодействието с полето за въвеждане, така че въвеждането да стане по-удобно.

![textfield](../../Resource/015_view.png)

В текущата реализация полето по подразбиране показва `0`. Това означава, че всеки път, когато потребителят иска да въведе сума, трябва първо да изтрие `0`, а след това да напише новата стойност, което не е удобно.

По-добре е полето първоначално да е празно, а не да показва `0`.

Причината е, че `TextField` и `number` са двупосочно обвързани:

```swift
@State private var number = 0
TextField("Amount", value: $number, format: .number)
```

Когато `number` се инициализира с `0`, полето също ще показва `0`.

Затова трябва да променим `number` в празна стойност, тоест `nil`.

```swift
@State private var number = nil
```

Тук ще се появи грешка:

```text
Generic parameter 'Value' could not be inferred
```

Това е точно случаят, който разгледахме в урока за „системата на типовете“: когато компилаторът не може автоматично да изведе типа на променлива, трябва да го декларираме явно.

Понеже задаваме `number` като `nil`, а самото `nil` не носи информация за тип, компилаторът не може да разбере дали променливата трябва да бъде `String`, `Int` или друг тип, и затова дава грешка.

Следователно трябва да декларираме типа на `number` изрично:

```swift
@State private var number:Int? = nil
```

Тук `Int?` означава Optional тип, което означава, че `number` може да бъде или `Int`, или `nil`.

Така, когато първоначално го инициализираме с `nil`, полето за въвеждане ще бъде празно вместо да показва `0`.

### Разопаковане на Optional тип

След като променим `number` в Optional тип, старият код за изчисление ще даде грешка:

```swift
amount += number
```

Съобщението ще бъде:

```text
Value of optional type 'Int?' must be unwrapped to a value of type 'Int'
```

Причината е, че `number` вече може да е `nil`. Ако директно участва в изчислението, това би било еквивалентно на:

```swift
amount += nil
```

Очевидно е, че `nil` не може да бъде събрано с `amount`, така че това не е валидна операция.

Затова, когато Optional тип участва в изчисление, първо трябва да бъде разопакован.

Тук можем да използваме `??`, за да зададем стойност по подразбиране:

```swift
amount += number ?? 0
```

Когато `number` е `nil`, ще бъде използвана стойността по подразбиране `0`; когато има стойност, ще бъде използвана реалната въведена стойност.

Така едновременно гарантираме коректността на изчислението и постигаме желаното поведение полето по подразбиране да е празно, без потребителят да трябва ръчно да изтрива `0`.

## Разширен код

```swift
struct ContentView: View {
    
    @AppStorage("amount") private var amount = 0
    @State private var number:Int? = nil
    
    var body: some View {
        VStack {
            Text("\(amount)")
                .font(.largeTitle)
                .fontWeight(.bold)
            TextField("Amount", value: $number, format: .number)
                .frame(width: 100)
                .padding(5)
                .border(Color.black, width: 1)
            Button("Save") {
                amount += number ?? 0
                number = 0
            }
            .buttonStyle(.borderedProminent)
        }
        .background {
            Image("1")
        }
    }
}
```
