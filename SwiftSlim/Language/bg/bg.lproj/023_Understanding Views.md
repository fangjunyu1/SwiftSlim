# Разбиране на изгледите

В този урок се връщаме отново към първоначалния `ContentView`.

```swift
struct ContentView: View {
    var body: some View {
        Text("Здравей, свят!")
    }
}
```

Когато за първи път се запознахме със SwiftUI, в `ContentView` имаше много ключови думи, които не разбирахме.

Например `struct`, `View`, `var`, `body`, `some View` и други ключови думи.

Тогава може би знаехме само: **ако напишем код в `body`, съдържанието ще се покаже на екрана.**

Сега вече сме учили `struct`, променливи, методи и протоколи, затова можем отново да разберем този код и така да си помогнем да разберем връзките между изгледите в SwiftUI.

## Да си припомним ContentView

След създаване на SwiftUI проект Xcode по подразбиране генерира файл `ContentView`.

Примерният код е следният:

```swift
import SwiftUI

struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
                .imageScale(.large)
                .foregroundStyle(.tint)
            Text("Здравей, свят!")
        }
        .padding()
    }
}
```

Този код може да се разбере, като го разделим на няколко части.

Първо:

```swift
import SwiftUI
```

Този ред означава импортиране на рамката SwiftUI.

След като импортираме SwiftUI, можем да използваме SwiftUI изгледи като `View`, `Text`, `Image`, `VStack`, `Button` и други.

## Структурата ContentView

Този ред код означава създаване на структура:

```swift
struct ContentView: View
```

Сред него:

```swift
struct ContentView
```

означава, че `ContentView` е структура.

В предишния урок разгледахме структурата `struct`; тя може да комбинира различни свързани полета заедно и може да съдържа свойства и методи.

Например:

```swift
struct ContentView: View {
    var name = "FangJunyu"

    func printName() {
        print(name)
    }

    var body: some View {
        Text("Здравей, свят!")
    }
}
```

В този пример `ContentView` съдържа три неща: `name`, `printName` и `body`.

С други думи, `struct` може да събере свързани свойства, методи и съдържание на изглед на едно място.

## View е протокол

Продължаваме да гледаме този ред код:

```swift
struct ContentView: View 
```

`View` след двоеточието означава: `ContentView` следва протокола `View`.

Можем да го разберем така: ако `ContentView` иска да стане SwiftUI изглед, той трябва да изпълни изискванията на протокола `View`.

### Протоколът Identifiable

Когато по-рано учихме списъка с филми, се запознахме с протокола `Identifiable`.

Тогава трябваше да обходим масив от `Movie` във `ForEach`, но структурата `Movie` нямаше уникален идентификатор, затова не поддържаше обхождане с `ForEach`.

В този момент трябваше да накараме структурата `Movie` да следва протокола `Identifiable`, за да отговори на изискването на `ForEach` за уникален идентификатор.

```swift
struct Movie: Identifiable {
    let id: Int
    let name: String
}
```

Протоколът `Identifiable` изисква: **този тип да има поле `id`, което го идентифицира.**

След като `Movie` предостави поле `id`, той може да изпълни изискването на `Identifiable`.

Така, когато го използваме във `ForEach`, SwiftUI може да разпознае всеки отделен запис.

```swift
ForEach(lists) { movie in
    // ...
}
```

### Протоколът View

Същото важи и за протокола `View`.

Ако един тип иска да стане SwiftUI изглед, той трябва да следва протокола `View`.

Най-важното изискване на протокола `View` е да предостави `body`.

```swift
struct ContentView: View {
    var body: some View {
        Text("Здравей, свят!")
    }
}
```

Можем да разберем изискването на протокола `View` така: **той приема само SwiftUI изгледа, който `body` връща, а `ContentView` показва SwiftUI изглед чрез `body`.**

### Какво става, ако не следваме View?

Ако напишем така:

```swift
struct ContentView {
}
```

Това е просто обикновена структура, а не SwiftUI изглед.

Тя няма директно да даде грешка, защото обикновената структура сама по себе си е валиден Swift код.

Но ако я поставим в предварителен преглед:

```swift
#Preview {
    ContentView()
}
```

тогава ще се появи грешка.

Причината е: **предварителният преглед трябва да показва SwiftUI изглед, а този `ContentView` не следва протокола `View`.**

Затова, ако искаме `ContentView` да може да се показва на интерфейса, трябва да го накараме да следва протокола `View`:

```swift
struct ContentView: View {
}
```

В същото време трябва да предоставим `body` според изискването на протокола `View`; ако няма `body`, компилаторът пак ще даде грешка.

Правилният запис е:

```swift
struct ContentView: View {
    var body: some View {
        Text("Здравей, свят!")
    }
}
```

Едва тогава `ContentView` е истински SwiftUI изглед, който може да се показва.

## body е входът за показване на изгледа

В SwiftUI `body` е входът към съдържанието на изгледа.

```swift
var body: some View {
    Text("Здравей, свят!")
}
```

Тук можем да го разделим на три части:

**1. var body**

```swift
var body
```

Означава дефиниране на променлива с име `body`.

**2. some View**

```swift
: some View
```

Означава, че този `body` ще върне изглед.

**3. { ... }**

```swift
{
    Text("Здравей, свят!")
}
```

Означава показваното съдържание на изгледа; в този код се показва изглед `Text`.

С други думи, каквото напишем вътре в `body`, това ще се покаже на интерфейса.

Например:

```swift
var body: some View {
    VStack(alignment: .leading) {
        Text("Име:")
            .fontWeight(.bold)
        Text("Fang Junyu")
    }
}
```

Този код означава: **на интерфейса се показва вертикално подредено съдържание с два текста.**

### Името body не може да се променя произволно

Трябва да внимаваме, че името `body` не може да се променя произволно.

Ако го напишем така:

```swift
struct ContentView: View {
    var content: some View {
        Text("Здравей, свят!")
    }
}
```

Този `content` също е свойство за изглед, но не може да замени `body`.

Защото името, което протоколът `View` изисква, е именно `body`.

## Какво е some View?

В SwiftUI `some View` означава връщане на конкретен изглед, само че не е нужно да пишем пълния тип на този изглед.

```swift
var body: some View
```

Тоест `some View` не означава, че няма тип; реалният тип все още съществува и компилаторът го знае, но този тип може да е много сложен, затова Swift ни позволява да използваме `some View`, за да го скрием.

### Какво е some?

В Swift `some` е ключова дума за деклариране на непрозрачен тип (opaque type).

Непрозрачен тип означава: **върнатата стойност има конкретен тип, компилаторът знае какъв е този конкретен тип, но ние не го изписваме директно.**

Например:

```swift
func makeView() -> some View {
    Text("Здравей")
}
```

Това означава, че `makeView` ще върне конкретен тип, който следва протокола `View`.

### Защо ни трябва some?

Когато учихме променливите, разбрахме: **стойностите в Swift трябва или изрично да имат отбелязан тип, или компилаторът автоматично да изведе типа им.**

Например:

```swift
let a: Int = 10
```

Тук константата `a` е от тип `Int`.

SwiftUI изгледите не са изключение. Когато използваме `body`, за да показваме SwiftUI изглед, трябва `body` да връща съответния SwiftUI тип.

Например, ако използваме `body`, за да показваме текстов изглед `Text`, трябва `body` да връща съответния тип `Text`.

```swift
var body: Text {
    Text("Fang Junyu")
}
```

В този пример вътре в `body` се показва изглед `Text`, а върнатият тип също е `Text`.

Само когато вътрешният тип и декларираният тип съвпадат, се изпълнява изискването на компилатора за съгласуваност на типовете.

**Проблем с несъответствие на типовете**

Ако зададем типа на `body` като `Color`, но показваме `Text`:

```swift
var body: Color {
    Text("Fang Junyu")
}
```

типът на `body` и типът на вътрешния код няма да съвпадат, което ще доведе до грешка за несъответствие на типовете.

```swift
Cannot convert return expression of type 'Text' to return type 'Color'
```

Тогава трябва ръчно да променим върнатия тип на `body`.

**Сложни типове изгледи**

В реалната разработка интерфейсът обикновено не съдържа само един `Text`.

Ако трябва да покажем по-сложен `VStack`, ще трябва ръчно да отбележим сложния върнат тип.

Например:

```swift
var body: VStack<TupleView<(Image,Text)>> {
    VStack {
        Image(systemName: "globe")
        Text("Здравей, свят!")
    }
}
```

Този код означава, че се връща `VStack`, а вътре в `VStack` има изгледи от тип `Image` и `Text`.

Върнатият тип на `body` е:

```swift
VStack<TupleView<(Image,Text)>>
```

Виждаме, че този върнат тип е много сложен и включва `TupleView`, който не сме срещали преди.

Тук `TupleView` може да се разбира като: **представяне на няколко подизгледа, обвити в един изглед.**

**Още по-сложни типове изгледи**

Ако добавим модификатори към изгледа:

```swift
var body: VStack<TupleView<(Image,Text)>> {
    VStack {
        Image(systemName: "globe")
            .imageScale(.large)
            .foregroundStyle(.tint)
        Text("Здравей, свят!")
    }
    .padding()
}
```

тогава върнатият тип на този изглед става много сложен и тук не можем да го опишем конкретно.

Съществуващият върнат тип `VStack<TupleView<(Image,Text)>>` вече няма да може да опише модификаторите и ще възникне грешка.

**some View опростява върнатия тип**

За тази ситуация SwiftUI предоставя кратко решение: използваме `some View`, за да опростим записа.

Когато променим `body` към тип `some View`:

```swift
var body: some View {
    VStack {
        Image(systemName: "globe")
            .imageScale(.large)
            .foregroundStyle(.tint)
        Text("Здравей, свят!")
    }
    .padding()
}
```

вече не е нужно при всяка промяна на кода на изгледа да изписваме пълния върнат тип.

Причината е, че `some View` означава връщане на конкретен изглед, който следва протокола `View`.

С други думи, SwiftUI изгледи като `VStack`, `Image` и други следват протокола `View`. След добавяне на модификатори върнатият тип на изгледа все още следва протокола `View`.

Затова всички те могат да бъдат върнати като `some View`.

### some View не може произволно да връща всякакъв изглед

Въпреки че `some View` може да скрива сложни типове, той не е "без тип".

Основното правило е: **компилаторът трябва да може да определи един конкретен върнат тип.**

Например:

```swift
var title: some View {
    Text("Здравей")
}
```

Тук се връща изглед `Text`.

Но ако се опитаме да върнем два изгледа, ще се появи грешка:

```swift
var title: some View {
    Text("Здравей")
    Image(systemName: "star")
}
```

Защото `some View` в крайна сметка трябва да върне една структура на изглед, а тук се опитваме да върнем два независими изгледа.

### Решение

Обикновено има две решения: контейнер и `@ViewBuilder`.

**1. Обвиване с контейнер**

```swift
var title: some View {
    VStack {
        Text("Здравей")
        Image(systemName: "star")
    }
}
```

В този код `title` връща един `VStack`, а `Text` и `Image` са подизгледи вътре във `VStack`.

Структурата на изгледа, която `some View` получава, е `VStack<...>`, така че тя отговаря на изискването на `some View` да се върне една структура на изглед. 

Това е и причината още от началото на изучаването на SwiftUI да препоръчваме да използваме контейнер `VStack` като външен слой, който обвива кода на изгледа.

**2. Използване на @ViewBuilder**

Можем също да добавим модификатора `@ViewBuilder` преди `some View`:

```swift
@ViewBuilder
var title: some View {
    Text("Здравей")
    Image(systemName: "star")
}
```

`@ViewBuilder` може да комбинира няколко изгледа в един върнат резултат.

По същество той отново обединява няколко изгледа в един изглед, за да изпълни изискването на `some View` за връщане на една структура на изглед.

### Защо body може директно да съдържа няколко изгледа?

Ако напишем няколко изгледа във `var body`, например:

```swift
var body: some View {
    Image(systemName: "globe")
        .imageScale(.large)
        .foregroundStyle(.tint)
    Text("Здравей, свят!")
}
```

в някои случаи такъв `body` няма да даде грешка.

Причината е, че `body` в протокола `View` по подразбиране поддържа `@ViewBuilder`:

```swift
@ViewBuilder @MainActor @preconcurrency var body: Self.Body { get }
```

В изходния код на протокола `View` можем да видим, че `@ViewBuilder` вече украсява променливата `body`.

Затова `body` има един допълнителен специален слой обработка спрямо обикновения `some View`, така че SwiftUI автоматично ни помага да комбинираме няколко изгледа в един.

**Препоръчва се използване на контейнери**

За да е по-лесно за начинаещите, препоръчваме първо да се използват контейнери като `VStack`, `HStack`, `ZStack`, `Group` за организиране на няколко изгледа.

Например:

```swift
var body: some View {
    VStack {
        Text("Здравей")
        Image(systemName: "star")
    }
}
```

Така кодът е по-интуитивен: **външният слой е `VStack`, а вътре има два подизгледа.**

## Разделяне на изгледа

Ако интерфейсът е сравнително прост, можем да напишем целия код в `body`.

Например:

```swift
struct ContentView: View {
    var body: some View {
        VStack(spacing: 30) {
            VStack(spacing: 20) {
                Text("Добре дошли в Qinote")
                Text("Добро утро")
            }
            HStack {
                Color.blue
                    .frame(width: 20, height: 10)
                    .cornerRadius(10)
                Color.red
                    .frame(width: 20, height: 10)
                    .cornerRadius(10)
                Color.green
                    .frame(width: 20, height: 10)
                    .cornerRadius(10)
            }
            Button("Натисни ме") {
                print("Натисни ме ")
            }
            .buttonStyle(.borderedProminent)
            Spacer()
        }
        .padding(20)
    }
}
```

Изгледът се показва така:

![view](../../Resource/023_view1.png)

Този код може да работи нормално, но ако кодът на изгледа стане много сложен, кодът в `body` ще става все по-дълъг, а трудността при четене и промяна постоянно ще се увеличава.

Тогава можем да разделим различните области като отделни свойства за изглед.

### Разделяне на горния изглед

Например, отделяме горната област:

```swift
var topHome: some View {
    HStack {
        Text("Начало")
            .font(.largeTitle)
        Spacer()
        Image(systemName: "bell")
    }
}
```

Тук `topHome` е променлива, която връща изглед.

Тъй като връща изглед, типът може да се запише като:

```swift
some View
```

Това всъщност е същата употреба като при `body`:

```swift
var topHome: some View { }
var body: some View { }
```

И двете използват променлива, за да показват изглед, а върнатият тип е `some View`.

### Разделяне на други изгледи

След това продължаваме да разделяме други части:

```swift
var welcomeText: some View {
    VStack(spacing: 20) {
        Text("Добре дошли в Qinote")
        Text("Добро утро")
    }
}

var colorList: some View {
    HStack {
        Color.blue
            .frame(width: 20, height: 10)
            .cornerRadius(10)
        Color.red
            .frame(width: 20, height: 10)
            .cornerRadius(10)
        Color.green
            .frame(width: 20, height: 10)
            .cornerRadius(10)
    }
}

var clickButton: some View {
    Button("Натисни ме") {
        print("Натисни ме")
    }
    .buttonStyle(.borderedProminent)
}
```

След разделянето можем да поставим променливите за изгледи в `body`:

```swift
struct ContentView: View {
    var body: some View {
        VStack(spacing: 30) {
            topBar
            welcomeText
            colorList
            clickButton
            Spacer()
        }
        .padding(20)
    }

    var topBar: some View {
        HStack {
            Text("Начало")
                .font(.largeTitle)
            Spacer()
            Image(systemName: "bell")
        }
    }

    var welcomeText: some View {
        VStack(spacing: 20) {
            Text("Добре дошли в Qinote")
            Text("Добро утро")
        }
    }

    var colorList: some View {
        HStack {
            Color.blue
                .frame(width: 20, height: 10)
                .cornerRadius(10)
            Color.red
                .frame(width: 20, height: 10)
                .cornerRadius(10)
            Color.green
                .frame(width: 20, height: 10)
                .cornerRadius(10)
        }
    }

    var clickButton: some View {
        Button("Натисни ме") {
            print("Натисни ме")
        }
        .buttonStyle(.borderedProminent)
    }
}
```

Сега `body` отговаря само за описване на структурата на страницата:

```swift
topBar
welcomeText
colorList
clickButton
```

Конкретната реализация на всяка част е поставена в съответното свойство по-долу.

Този начин на разделяне на изгледа прави структурата на страницата по-ясна; всяка област има свое име и се чете по-интуитивно.

Когато променяме определена част от кода, можем директно да намерим съответната област на изгледа, вместо да търсим в куп сложен код.

### some View връща един изглед

В този разделен изглед, ако се опитаме да върнем няколко изгледа:

```swift
var topBar: some View {
    Text("Начало")
        .font(.largeTitle)
    Spacer()
    Image(systemName: "bell")
}
```

ще се появи грешка, защото `some View` може да върне само една структура на изглед.

За разлика от `body`, обикновените изчисляеми свойства по подразбиране нямат `@ViewBuilder`.

Затова, ако трябва да върнем няколко изгледа, можем да използваме контейнер `VStack`, `HStack` или `Group`, за да комбинираме няколко изгледа в един.

```swift
var topBar: some View {
    VStack {
        Text("Начало")
            .font(.largeTitle)
        Spacer()
        Image(systemName: "bell")
    }
}
```

Или да добавим `@ViewBuilder`, за да комбинираме изгледите:

```swift
@ViewBuilder
var topBar: some View {
    Text("Начало")
        .font(.largeTitle)
    Spacer()
    Image(systemName: "bell")
}
```

Така можем да реализираме връщане на няколко изгледа.

### if връща няколко изгледа

Например:

```swift
var topHome: some View {
    if step == "Начало" {
        HStack {
            Text("Начало")
                .font(.largeTitle)
            Spacer()
            Image(systemName: "bell")
        }
    } else {
        Text("Няма")
    }
}
```

В този код `topHome` според съдържанието на `step` връща два различни типа изгледи: `HStack` или `Text`.

Тъй като `some View` се нуждае от един определен конкретен върнат тип, двата клона на `if` в обикновено изчисляемо свойство не могат директно да връщат различни типове изгледи, което води до грешка.

**Решението все още е да използваме контейнер за обвиване или да използваме `@ViewBuilder`.**

### Защо при разделените изгледи не използваме скоби?

Когато използваме отделените свойства за изглед в `body`, пишем директно:

```swift
topBar
welcomeText
colorList
clickButton
```

а не:

```swift
topBar()
welcomeText()
colorList()
clickButton()
```

Причината е, че `topBar`, `welcomeText`, `colorList` и `clickButton` са променливи; тези променливи всъщност са изчисляеми свойства, а не методи.

**Свойствата не се използват със скоби.**

Ако ги напишем като метод, също можем да постигнем подобен ефект:

```swift
func topBar() -> some View {
    HStack {
        Text("Начало")
            .font(.largeTitle)
        Spacer()
        Image(systemName: "bell")
    }
}
```

Този код също може да постигне подобен ефект; извикването на този метод ще върне изглед.

При използване трябва да се напише така:

```swift
topBar()
```

В SwiftUI, ако просто разделяме изглед, който не се нуждае от входни параметри, по-често се използва изчисляемо свойство.

```swift
var topBar: some View { ... }
```

## Вложени изгледи

Освен да разделяме кода на свойства, можем също да създаваме нови структури за изгледи.

Например:

```swift
struct PinkColorView: View {
    var body: some View {
        Color.pink
            .frame(width: 80, height: 40)
            .shadow(radius: 3)
    }
}
```

Този `PinkColorView`, както и `ContentView`, е независим SwiftUI изглед.

Ако трябва да го покажем в `ContentView`, можем да добавим `()` след името на изгледа.

Например:

```swift
struct ContentView: View {
    var body: some View {
        PinkColorView()
    }
}
```

Тук:

```swift
PinkColorView()
```

означава създаване на инстанция на `PinkColorView`.

Тъй като `PinkColorView` следва протокола `View`, той може да бъде поставен за показване в други изгледи, точно както `Text`, `Image` и `Button`.

```swift
struct ContentView: View {
    var body: some View {
        PinkColorView()

        Text("PinkColorView")
    }
}
```

Това е вложен изглед: един изглед може да се показва вътре в друг изглед.

## Свойство за изглед или нов изглед?

Когато кодът на изгледа е сравнително прост, обикновено няма нужда да го разделяме; само когато кодът стане по-сложен, е препоръчително да разделяме изгледа.

### Сценарии за използване на свойства за изглед

Ако това е само малка част от текущия изглед, можем да използваме свойство за изглед.

Например:

```swift
var topBar: some View { ... }
var clickButton: some View { ... }
```

Такова съдържание обикновено се използва само в текущия изглед.

### Сценарии за създаване на нов изглед

Ако това съдържание е сравнително независимо или в бъдеще може да се използва повторно в няколко страници, тогава е по-подходящо да създадем нов изглед.

Например:

```swift
Text("Qinote")
    .padding()
    .background(.blue.opacity(0.1))
    .cornerRadius(12)
```

Това е заглавен изглед. Ако искаме да използваме този изглед като общ компонент и да го преизползваме в други изгледи.

Можем да създадем нов изглед:

```swift
struct CourseCardView: View {
    var body: some View {
        Text("Qinote")
            .padding()
            .background(.blue.opacity(0.1))
            .cornerRadius(12)
    }
}
```

След това го използваме на други места:

```swift
struct SettingsView: View {
    var body: some View {
        CourseCardView()
    }
}
struct ContentView: View {
    var body: some View {
        CourseCardView()
    }
}
```

Тогава този изглед е независим компонент, който може да се използва повторно.

Можем просто да го обобщим така:

- Област със съдържание в текущия изглед: може да се използва `var xxx: some View`
- Независим компонент, който може да се преизползва: може да се създаде нов `struct XxxView: View`
- Компонент, който трябва да получава данни: обикновено е по-подходящо да се създаде нов изглед.

## Обобщение

В този урок отново разбрахме най-основната, но и най-важна структура на изгледите в SwiftUI.

`ContentView` е структура, дефинирана чрез `struct`, която се използва за организиране на кода, свързан със страницата.

`View` е протокол; ако една структура иска да стане SwiftUI изглед, тя трябва да следва протокола `View`.

`body` е съдържанието, което протоколът `View` изисква да предоставим, и то определя какво показва този изглед.

`some View` означава връщане на определен конкретен изглед; не е нужно да изписваме неговия сложен пълен тип.

Когато кодът на изгледа стане дълъг, можем да използваме `var xxx: some View`, за да разделим кода на изгледа на няколко малки области.

Когато дадена част от съдържанието е сравнително независима или трябва да се използва повторно, можем да създадем нова структура за изглед и да я покажем в други изгледи чрез `XxxView()`.

Казано просто, интерфейсът в SwiftUI е съставен от отделни изгледи, комбинирани заедно; една страница може да съдържа няколко малки изгледа, а един малък изглед може да съдържа други изгледи.
