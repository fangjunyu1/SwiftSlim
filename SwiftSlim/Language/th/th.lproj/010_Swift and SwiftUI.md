# Swift และ SwiftUI

เมื่อเรียนการพัฒนาบนแพลตฟอร์มของ Apple บทเรียนจำนวนมากมักอธิบาย Swift และ SwiftUI แยกจากกัน ซึ่งทำให้ผู้เริ่มต้นรู้สึกว่าเนื้อหาถูกตัดขาดออกจากกัน และอาจเข้าใจผิดว่าทั้งสองเป็นเทคโนโลยีที่แยกจากกันโดยสิ้นเชิง

ในความเป็นจริง Swift และ SwiftUI เป็นชุดเทคโนโลยีที่ผสานกันอย่างแนบแน่น: Swift คือภาษาโปรแกรม ทำหน้าที่จัดการตรรกะและข้อมูล; ส่วน SwiftUI คือเฟรมเวิร์กสำหรับสร้างส่วนติดต่อผู้ใช้ที่สร้างขึ้นบน Swift โดยทำหน้าที่เรนเดอร์มุมมองและจัดการการโต้ตอบ

บทเรียนนี้จะช่วยให้คุณสร้างกรอบความเข้าใจที่ชัดเจนว่า Swift คืออะไร SwiftUI คืออะไร และทั้งสองทำงานร่วมกันในโค้ดอย่างไร

## Swift คืออะไร?

Swift คือภาษาโปรแกรมสมัยใหม่ที่ Apple เปิดตัวขึ้น ใช้สำหรับพัฒนาแอปบนแพลตฟอร์ม iOS, macOS, watchOS และ tvOS

![swift](../../RESOURCE/010_swift.png)

Swift เป็นภาษาที่มี strong typing และเป็น static language โดยให้ความสำคัญกับความปลอดภัยมากขึ้น ช่วยลดข้อผิดพลาดที่พบบ่อยในการเขียนโปรแกรมได้จำนวนมาก (เช่น null pointer, การเข้าถึงเกินขอบเขต เป็นต้น) และรองรับการทำงานร่วมกับ C และ Objective-C ได้อย่างไร้รอยต่อ

ในการพัฒนาจริง Swift มักรับผิดชอบส่วนของตรรกะในแอปมากกว่า เช่น การประมวลผลข้อมูล การร้องขอเครือข่าย การจัดเก็บข้อมูล เป็นต้น

## SwiftUI คืออะไร？

SwiftUI คือเฟรมเวิร์กใหม่สำหรับสร้างหน้าจอที่ Apple เปิดตัวในปี 2019 โดย SwiftUI ใช้แนวคิดการเขียนโปรแกรมแบบ declarative ซึ่งช่วยให้นักพัฒนาสามารถใช้โค้ดที่กระชับกว่าเดิมในการอธิบายโครงสร้างของหน้าจอและพฤติกรรมการโต้ตอบ

![swiftui](../../RESOURCE/010_swiftui.png)

การเขียนโปรแกรมแบบ declarative หมายความว่า นักพัฒนาเพียงแค่บอกระบบว่าต้องการให้แสดงอะไร ระบบก็จะรีเฟรชมุมมองตามการเปลี่ยนแปลงของข้อมูลโดยอัตโนมัติ โดยไม่จำเป็นต้องอัปเดตสถานะของหน้าจอด้วยตนเอง ซึ่งช่วยลดความซับซ้อนของการพัฒนา UI ได้อย่างมาก

แกนหลักของ SwiftUI คือคอมโพเนนต์ของมุมมอง (เช่น `Text`, `Image`, `Button` เป็นต้น) และคอนเทนเนอร์สำหรับจัดวางเลย์เอาต์ (เช่น `VStack`, `HStack`, `ZStack`) คอมโพเนนต์เหล่านี้โต้ตอบกันผ่าน data binding และการจัดการสถานะ ทำให้หน้าจอสามารถตอบสนองต่อการเปลี่ยนแปลงของข้อมูลและรีเฟรชได้โดยอัตโนมัติ

## Swift และ SwiftUI ทำงานร่วมกันอย่างไร？

หน้าที่ของ Swift และ SwiftUI สามารถสรุปได้ดังนี้:

**1. Swift: จัดการตรรกะและข้อมูล**

Swift ใช้หลัก ๆ สำหรับจัดการข้อมูล เก็บสถานะ และดำเนินการตามตรรกะ โค้ดเหล่านี้โดยทั่วไปไม่ได้ส่งผลต่อการเรนเดอร์หน้าจอโดยตรง แต่มีหน้าที่จัดการข้อมูลและพฤติกรรมของแอป

ตัวอย่างเช่น การประมวลผลข้อมูลใน Swift:

```swift id="1poqs6"
let markdown = try? String(contentsOf: url)
```

โค้ดเหล่านี้เกี่ยวข้องกับการประกาศตัวแปร การกำหนดค่าคงที่ ฟังก์ชัน control flow และอื่น ๆ โดยไม่ได้ส่งผลต่อการแสดงผลของวิวโดยตรง

**2. SwiftUI: ประกาศและเรนเดอร์หน้าจอ**

ส่วน SwiftUI ใช้สำหรับสร้างเลย์เอาต์และเนื้อหาของส่วนติดต่อผู้ใช้ โดยมีหน้าที่หลักคือเรนเดอร์หน้าจอและจัดการการโต้ตอบของวิว

ตัวอย่างเช่น การสร้างมุมมองข้อความด้วย SwiftUI:

```swift id="z3tpw6"
Text("SwiftSlim")
```

โค้ดเหล่านี้ใช้สำหรับสร้างและควบคุมองค์ประกอบของหน้าจอ และมีปฏิสัมพันธ์กับ UI โดยตรง

### ตัวอย่างพื้นฐาน

ด้านล่างคือตัวอย่าง SwiftUI แบบง่าย:

```swift id="x3wxga"
// ContentView.swift

import SwiftUI

struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
                .imageScale(.large)
                .foregroundStyle(.tint)
            Text("Hello, world!")
        }
        .padding()
    }
}

#Preview {
    ContentView()
}
```

ในตัวอย่างนี้:

**โค้ด SwiftUI**: `VStack`, `Image`, `Text`, `padding` ล้วนเป็นส่วนของ SwiftUI ทำหน้าที่แสดงผลและจัดวางหน้าจอ

**โค้ด Swift**: คอมเมนต์ `//` ที่ด้านบนของไฟล์ และ `import SwiftUI` เป็นส่วนหนึ่งของโครงสร้างโค้ด Swift; ส่วน `#Preview` ที่ด้านล่างเป็น macro สำหรับใช้ดูตัวอย่างใน Xcode และไม่ได้มีส่วนร่วมในการเรนเดอร์หน้าจอจริง

### ตัวอย่างขั้นสูงขึ้น

ในโปรเจกต์จริง Swift และ SwiftUI มักถูกใช้ร่วมกัน เพื่อทำงานในส่วนที่แต่ละฝ่ายถนัด:

```swift id="5noczy"
struct ContentView: View {
    @State private var name = "FangJunyu"

    func printName() {
        print("name: \(name)")
    }

    var body: some View {
        VStack {
            Text(name)
            Button("Print") {
                printName()
            }
        }
        .padding()
    }
}
```

ในตัวอย่างนี้:

**โค้ด SwiftUI**: `@State` คือ property wrapper เฉพาะของ SwiftUI ใช้สำหรับประกาศสถานะที่เปลี่ยนแปลงได้ของวิว; `Button` และ `Text` คือคอมโพเนนต์มุมมองของ SwiftUI ใช้สำหรับแสดงผลหน้าจอและโต้ตอบกับผู้ใช้

**โค้ด Swift**: `private var name` และ `func printName()` เป็นโค้ด Swift ใช้สำหรับจัดเก็บข้อมูลและดำเนินการตามตรรกะ โดยไม่ได้ส่งผลต่อการเรนเดอร์วิวโดยตรง

เมื่อผู้ใช้กดปุ่ม SwiftUI จะเรียกโค้ดการทำงานภายในปุ่ม:

```swift id="iq91km"
Button("Print") {
    printName()
}
```

ในที่นี้ `Button` คือคอมโพเนนต์ของ SwiftUI ส่วนฟังก์ชัน `printName()` ที่ถูกเรียกใช้คือโค้ด Swift ซึ่งรับผิดชอบตรรกะการทำงานจริง

ความร่วมมือนี้ทำให้ Swift และ SwiftUI สามารถทำงานร่วมกันได้อย่างไร้รอยต่อ โดย Swift รับผิดชอบการจัดการข้อมูลและตรรกะ ส่วน SwiftUI รับผิดชอบการแสดงผลส่วนติดต่อผู้ใช้

## โดยทั่วไปโค้ด Swift และ SwiftUI เขียนไว้ตรงไหน?

ใน SwiftUI หน้าจอถูกสร้างผ่านวิวที่คืนค่าจาก property `body` ดังนั้นโค้ดทั้งหมดที่ใช้บรรยายหน้าจอมักจะถูกเขียนไว้ภายใน `body`

ตัวอย่างเช่น:

```swift id="bm6dwh"
struct ContentView: View {
    var body: some View {
        Text("Hello")
    }
}
```

ในตัวอย่างนี้ `Text` เป็นคอมโพเนนต์มุมมองของ SwiftUI ดังนั้นจึงต้องเขียนไว้ใน `body` เพราะ SwiftUI จะอ่าน `body` เพื่อสร้างหน้าจอ

ส่วนโค้ดที่ไม่เกี่ยวกับหน้าจอ เช่น ตัวแปร ฟังก์ชัน หรือตรรกะการประมวลผลข้อมูล มักจะเขียนไว้นอก `body` ตัวอย่างเช่น:

```swift id="gq2t9q"
struct ContentView: View {

    // Swift：ข้อมูลหรือตรรกะ
    var name = "Fang"

    func printName() {
        print(name)
    }

    // SwiftUI：หน้าจอ
    var body: some View {
        Text(name)
    }
}
```

อย่างไรก็ตาม ต้องสังเกตว่าใน `body` เองก็ยังสามารถใช้ไวยากรณ์ของ Swift ได้ เช่น `if`, `for` และคำสั่งควบคุมการทำงานอื่น ๆ เพียงแต่มันถูกใช้เพื่อควบคุมการสร้างมุมมองของ SwiftUI:

```swift id="agxvam"
if isLogin {
    Text("Welcome")
}
```

ดังนั้น ในการพัฒนา SwiftUI สามารถเข้าใจแบบง่าย ๆ ได้ว่า: โค้ดของวิว (`Text`, `Image`, `Button` เป็นต้น) มักเขียนไว้ใน `body`; ส่วนโค้ดด้านข้อมูลและตรรกะ (ตัวแปร ฟังก์ชัน เป็นต้น) มักเขียนไว้นอก `body`

## ไฟล์ Swift

เมื่อเรียนลึกขึ้น เราจะได้พบกับสถาปัตยกรรม MVVM ซึ่งในชั้น ViewModel และ Model มักประกอบด้วยโค้ด Swift ล้วน และแยกออกจากชั้นมุมมอง (SwiftUI) อย่างชัดเจน

ตัวอย่างเช่น คลาสที่ใช้จัดการสถานะของแอป:

```swift id="mdbdqw"
class AppStorage: ObservableObject {
    static var shared = AppStorage()
    private var isLoading = false
}
```

หรือโครงสร้างที่ใช้อธิบายข้อมูล:

```swift id="s24n37"
struct SponsorStruct: Identifiable {
    var id:String
    var icon: String
}
```

หน้าที่ของไฟล์ประเภทนี้คือการจัดการและเก็บข้อมูลอย่างถาวร ไม่ได้แสดงผลบนหน้าจอโดยตรง ดังนั้นจึงถือว่าเป็นส่วนของ Swift ทั้งหมด

## ภูมิหลังทางประวัติศาสตร์

การจะเข้าใจความสัมพันธ์ระหว่าง Swift กับ SwiftUI ได้ดี ยังต้องเข้าใจพัฒนาการของเทคโนโลยีสำหรับการพัฒนาของ Apple ด้วย ณ ปี 2026 ชุดเทคโนโลยีนี้ผ่านการพัฒนามาหลายรอบแล้ว

### ประวัติของ Swift

ก่อนที่ Swift จะปรากฏขึ้น Objective-C คือภาษาโปรแกรมหลักของแพลตฟอร์ม Apple และรองรับการเขียนผสมกับภาษา C ไวยากรณ์ของมันค่อนข้างยาวและมีความซับซ้อน ทำให้ผู้เริ่มต้นเรียนรู้ได้ยากกว่า:

```Obj-c
// รูปแบบของ Objective-C
NSString *name = @"Fang";
NSLog(@"Hello，%@", name);
```

ในปี 2014 Apple เปิดตัวภาษา Swift ในงาน WWDC Swift มีไวยากรณ์ที่ทันสมัยกว่า มีความปลอดภัยด้านประเภทข้อมูลสูงกว่า และค่อย ๆ เข้ามาแทนที่ Objective-C จนกลายเป็นภาษาหลักในการพัฒนา:

```swift id="ozneci"
// รูปแบบของ Swift
let name = "Fang"
print("Hello, \(name)")
```

อย่างไรก็ตาม Objective-C ไม่ได้หายไปจากประวัติศาสตร์ทันที มันยังคงถูกใช้อย่างแพร่หลายในโปรเจกต์เดิมจำนวนมากและในเฟรมเวิร์กระดับล่าง การเข้าใจไวยากรณ์พื้นฐานของมันยังคงมีคุณค่า สำหรับการดูแลโปรเจกต์เก่าและการทำความเข้าใจระบบระดับล่าง

### ประวัติของ SwiftUI

ก่อนการมาถึงของ SwiftUI ฝั่ง iOS ใช้ **UIKit** และฝั่ง macOS ใช้ **AppKit** เฟรมเวิร์กทั้งสองนี้ใช้แนวทางการเขียนโปรแกรมแบบ “imperative” นักพัฒนาต้องลากวางคอนโทรลผ่าน Storyboard หรือเขียนโค้ดด้วยตนเองเพื่อควบคุมสถานะของวิว ทำให้มีปริมาณโค้ดมาก ต้นทุนการดูแลรักษาสูง และจะยิ่งชัดเจนเมื่อหน้าจอมีตรรกะซับซ้อน

![storyboard](../../RESOURCE/010_xcode.png)

ในปี 2019 Apple เปิดตัว SwiftUI อย่างเป็นทางการในงาน WWDC SwiftUI ได้นำแนวคิดการเขียนโปรแกรมแบบ “declarative” เข้ามาใช้ และลดความซับซ้อนของกระบวนการพัฒนา UI ลงอย่างมาก

![storyboard](../../RESOURCE/010_xcode1.png)

สิ่งที่ควรสังเกตคือ SwiftUI ไม่ได้เป็นการทำงานระดับล่างที่แยกออกมาอย่างสมบูรณ์ บนแต่ละแพลตฟอร์ม มันมีลักษณะเป็นการ bridge และทำงานร่วมกับ UIKit (iOS) หรือ AppKit (macOS) โดยระดับล่างยังคงอาศัยสองเฟรมเวิร์กนี้ในการทำงาน

### ความสัมพันธ์ระหว่าง Swift กับ UIKit/AppKit

แม้ว่า Swift จะเป็นภาษาโปรแกรมอเนกประสงค์ที่สามารถทำงานบนแพลตฟอร์มต่าง ๆ ของ Apple ได้ แต่มันก็ไม่สามารถแทนที่ UIKit หรือ AppKit ได้ทั้งหมด สำหรับความต้องการด้าน UI ที่ซับซ้อนบางอย่าง หรือความสามารถบางจุดที่ SwiftUI ยังไม่ครอบคลุม ก็ยังจำเป็นต้องอาศัย UIKit หรือ AppKit มาช่วยทำงาน

ตัวอย่างเช่น UIKit มีความสมบูรณ์และผ่านการใช้งานจริงมาอย่างยาวนานในด้านการจัดการ view controller ที่ซับซ้อน เอฟเฟกต์แอนิเมชัน การรู้จำ gesture และงานลักษณะอื่น ๆ แม้ว่า SwiftUI จะพัฒนาความสามารถในด้านเหล่านี้อย่างต่อเนื่อง แต่ในบางกรณีขอบ ๆ ก็ยังมีข้อจำกัดอยู่

ดังนั้น นักพัฒนาจำนวนมากจึงยังคงใช้ SwiftUI ร่วมกับ UIKit (หรือ AppKit) ภายในโปรเจกต์เดียวกัน เพื่อใช้ประโยชน์จากข้อดีของทั้งสองฝั่งอย่างเต็มที่

หากมองจากมุมนี้ SwiftUI สามารถเข้าใจได้ว่าเป็นการห่อหุ้มระดับสูงรูปแบบหนึ่งของ UIKit / AppKit ขณะเรียน SwiftUI การเข้าใจแนวคิดพื้นฐานของ UIKit และ AppKit ไปพร้อมกันบ้าง จะช่วยให้สามารถตัดสินใจทางเทคนิคได้เหมาะสมยิ่งขึ้นเมื่อต้องดูแลโปรเจกต์เก่าหรือสร้างฟังก์ชันที่ซับซ้อน

## สรุป

**Swift**: ใช้หลัก ๆ สำหรับเขียนตรรกะ การประมวลผลข้อมูล control flow และส่วนที่ไม่เกี่ยวกับการจัดวางหน้าจอ

**SwiftUI**: ใช้สำหรับสร้างส่วนติดต่อผู้ใช้แบบ declarative โค้ดที่เกี่ยวกับเนื้อหาและเลย์เอาต์ของวิวถือเป็นส่วนของ SwiftUI

ในการพัฒนาจริง Swift และ SwiftUI มักถูกใช้ร่วมกัน โดย Swift รับผิดชอบตรรกะ ส่วน SwiftUI รับผิดชอบหน้าจอ

ตั้งแต่ Objective-C และ UIKit มาจนถึง Swift และ SwiftUI สภาพแวดล้อมการพัฒนาของ Apple ค่อย ๆ เปลี่ยนไปสู่รูปแบบการเขียนโปรแกรมที่ทันสมัยและกระชับมากขึ้น แต่ UIKit และ AppKit ก็ยังคงมีความสำคัญและไม่อาจมองข้ามได้ในโปรเจกต์แบบดั้งเดิมจำนวนมาก

เมื่อเข้าใจความสัมพันธ์ระหว่าง Swift และ SwiftUI แล้ว เราจะสามารถพัฒนา iOS/macOS ได้อย่างมีประสิทธิภาพมากขึ้น และยังสามารถตัดสินใจทางเทคนิคได้เหมาะสมยิ่งขึ้นเมื่อต้องดูแลโปรเจกต์เก่าด้วย
