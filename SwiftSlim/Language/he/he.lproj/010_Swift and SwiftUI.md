# Swift ו-SwiftUI

כאשר לומדים לפתח עבור הפלטפורמות של Apple, הרבה מדריכים נוטים להסביר את Swift ואת SwiftUI בנפרד. זה עלול לגרום למתחילים להרגיש שמדובר בשתי טכנולוגיות נפרדות לחלוטין.

בפועל, Swift ו-SwiftUI הם שילוב הדוק מאוד: Swift היא שפת התכנות, שאחראית על הלוגיקה ועל הנתונים; ו-SwiftUI היא מסגרת לבניית ממשקי משתמש, שנבנתה על בסיס Swift, ואחראית על הרינדור והאינטראקציה של התצוגות.

השיעור הזה יעזור לבנות מסגרת מחשבה ברורה: מהי Swift, מהי SwiftUI, ואיך הן משתפות פעולה בתוך הקוד.

## מהי Swift?

Swift היא שפת תכנות מודרנית שפותחה על ידי Apple, ומשמשת לפיתוח אפליקציות עבור iOS, macOS, watchOS ו-tvOS.

![swift](../../RESOURCE/010_swift.png)

Swift היא שפה חזקה מבחינת טיפוסים (`strongly typed`) וסטטית, עם דגש משמעותי על בטיחות, כדי לצמצם הרבה שגיאות תכנות נפוצות (כמו null pointers, חריגה מגבולות ועוד). היא גם תומכת באינטגרציה חלקה עם C ו-Objective-C.

בפיתוח מעשי, Swift אחראית יותר על הצד הלוגי של האפליקציה, כמו עיבוד נתונים, בקשות רשת, פעולות אחסון ועוד.

## מהי SwiftUI?

SwiftUI היא מסגרת חדשה לבניית ממשקים, שהוצגה על ידי Apple בשנת 2019. SwiftUI משתמשת בגישת תכנות דקלרטיבית (`declarative`), שמאפשרת למפתחים לתאר את מבנה הממשק וההתנהגות שלו באמצעות קוד קצר וברור יותר.

![swiftui](../../RESOURCE/010_swiftui.png)

תכנות דקלרטיבי אומר שהמפתח רק מתאר למערכת מה הוא רוצה שיוצג, והמערכת כבר מרעננת את התצוגה אוטומטית כאשר הנתונים משתנים. כבר לא צריך לעדכן ידנית את מצב הממשק, וזה מפשט מאוד את פיתוח ה-UI.

הליבה של SwiftUI מורכבת מרכיבי תצוגה (`Text`、`Image`、`Button` ועוד) וממיכלי פריסה (`VStack`、`HStack`、`ZStack`). רכיבים אלה עובדים יחד דרך binding של נתונים וניהול state, כך שהממשק מגיב לשינויים בנתונים ומתעדכן אוטומטית.

## איך Swift ו-SwiftUI משתפות פעולה?

אפשר לסכם את האחריות של Swift ושל SwiftUI כך:

**1. Swift: מטפלת בלוגיקה ובנתונים**

Swift משמשת בעיקר לניהול נתונים, לשמירת state ולהרצת לוגיקה. קטעי קוד כאלה בדרך כלל לא משפיעים ישירות על הרינדור של הממשק, אלא אחראים על הנתונים ועל ההתנהגות.

למשל, עיבוד נתונים ב-Swift:

```swift
let markdown = try? String(contentsOf: url)
```

קטעי קוד כאלה כוללים הצהרת משתנים, הגדרת קבועים, פונקציות וזרימות בקרה, והם לא מציגים ישירות ממשק.

**2. SwiftUI: מצהירה ומרנדרת את הממשק**

SwiftUI משמשת לבניית הפריסה והתוכן של ממשק המשתמש, ואחראית בעיקר על הרינדור ועל האינטראקציה.

למשל, יצירת תצוגת טקסט ב-SwiftUI:

```swift
Text("SwiftSlim")
```

כל אלה הם קטעי קוד שנועדו לבנות ולהפעיל רכיבי ממשק, והם מקיימים אינטראקציה ישירה עם ממשק המשתמש.

### דוגמה בסיסית

הנה דוגמה פשוטה ל-SwiftUI:

```swift
// ContentView.swift

import SwiftUI

struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
                .imageScale(.large)
                .foregroundStyle(.tint)
            Text("Hello, world!")
        }
        .padding()
    }
}

#Preview {
    ContentView()
}
```

בדוגמה הזו:

**קוד SwiftUI**: `VStack`、`Image`、`Text` ו-`padding` כולם שייכים ל-SwiftUI, ואחראים על התצוגה והפריסה של הממשק.

**קוד Swift**: ההערה `//` בחלק העליון של הקובץ ו-`import SwiftUI` שייכים למבנה הקוד של Swift; ובתחתית, `#Preview` הוא מאקרו של Xcode לתצוגה מקדימה, שאינו משתתף ברינדור בפועל של הממשק.

### דוגמה מתקדמת יותר

בפרויקטים אמיתיים, Swift ו-SwiftUI משמשות לעיתים קרובות יחד כדי שכל אחת מהן תבצע את מה שהיא טובה בו:

```swift
struct ContentView: View {
    @State private var name = "FangJunyu"

    func printName() {
        print("name: \(name)")
    }

    var body: some View {
        VStack {
            Text(name)
            Button("Print") {
                printName()
            }
        }
        .padding()
    }
}
```

בדוגמה הזאת:

**קוד SwiftUI**: `@State` הוא property wrapper מיוחד של SwiftUI, שמשמש להצהרה על מצב ניתן לשינוי של התצוגה; `Button` ו-`Text` הם רכיבי תצוגה של SwiftUI, שאחראים על ההצגה ועל האינטראקציה עם המשתמש.

**קוד Swift**: `private var name` ו-`func printName()` הם קוד Swift רגיל, שמשמש לשמירת נתונים ולהפעלת לוגיקה, ואינו אחראי ישירות על הרינדור של התצוגה.

כאשר המשתמש לוחץ על הכפתור, SwiftUI מפעיל את קוד הפעולה שבתוך הכפתור:

```swift
Button("Print") {
    printName()
}
```

כאן `Button` הוא רכיב של SwiftUI, בעוד שהפונקציה `printName()` היא קוד Swift שמטפל בלוגיקה עצמה.

שיתוף הפעולה הזה מאפשר ל-Swift ול-SwiftUI להתחבר בצורה חלקה: Swift מטפלת בנתונים ובלוגיקה, ו-SwiftUI מטפלת בהצגת הממשק.

## איפה בדרך כלל כותבים קוד של Swift ושל SwiftUI?

ב-SwiftUI, הממשק נבנה מתוך התצוגה שמוחזרת על ידי המאפיין `body`. לכן, כל הקוד שמתאר את הממשק נכתב בדרך כלל בתוך `body`.

למשל:

```swift
struct ContentView: View {
    var body: some View {
        Text("Hello")
    }
}
```

בדוגמה הזאת, `Text` הוא רכיב תצוגה של SwiftUI, ולכן הוא חייב להיות כתוב בתוך `body`, כי SwiftUI בונה את הממשק מתוך מה ש-`body` מחזיר.

לעומת זאת, קוד שאינו קשור ישירות לממשק, כמו משתנים, פונקציות או לוגיקת עיבוד נתונים, נכתב בדרך כלל מחוץ ל-`body`.

למשל:

```swift
struct ContentView: View {

    // Swift: נתונים או לוגיקה
    var name = "Fang"

    func printName() {
        print(name)
    }

    // SwiftUI: ממשק
    var body: some View {
        Text(name)
    }
}
```

חשוב לשים לב שגם בתוך `body` עדיין אפשר להשתמש בתחביר של Swift, למשל `if`、`for` ועוד, אבל הם רק עוזרים לשלוט באופן שבו תצוגות SwiftUI נוצרות:

```
if isLogin {
    Text("Welcome")
}
```

לכן, בפיתוח עם SwiftUI אפשר להבין את זה כך: קוד תצוגה (`Text`、`Image`、`Button` ועוד) נכתב בדרך כלל בתוך `body`; וקוד של נתונים ולוגיקה (משתנים, פונקציות ועוד) נכתב בדרך כלל מחוץ ל-`body`.

## קבצי Swift

ככל שמעמיקים בלימוד, ניתקל גם בארכיטקטורת MVVM, שבה שכבות ה-ViewModel וה-Model מורכבות בדרך כלל מקוד Swift טהור, ונפרדות לחלוטין משכבת התצוגה (`SwiftUI`).

למשל, מחלקה שמנהלת את מצב האפליקציה:

```swift
class AppStorage: ObservableObject {
    static var shared = AppStorage()
    private var isLoading = false
}
```

או מבנה שמתאר מודל נתונים:

```
struct SponsorStruct: Identifiable {
    var id:String
    var icon: String
}
```

התפקיד של קבצים כאלה הוא לנהל ולשמר נתונים, ולא להציג אותם ישירות בממשק, ולכן הם שייכים לחלוטין לעולם של Swift.

## רקע היסטורי

כדי להבין את הקשר בין Swift ל-SwiftUI, צריך גם להבין את תהליך ההתפתחות של טכנולוגיות הפיתוח של Apple. נכון לשנת 2026, ה-stack הזה עבר כמה וכמה איטרציות.

### ההיסטוריה של Swift

לפני הופעת Swift, Objective-C הייתה שפת התכנות המרכזית של פלטפורמות Apple, והיא תמכה גם בשילוב עם שפת C. התחביר שלה היה ארוך יותר, ולמתחילים היה קשה יותר להיכנס אליה:

```Obj-c
// כתיבה ב-Objective-C
NSString *name = @"Fang";
NSLog(@"Hello，%@", name);
```

בשנת 2014, Apple הכריזה ב-WWDC על שפת Swift. התחביר של Swift מודרני יותר, בטיחות הטיפוסים בה גבוהה יותר, והיא החליפה בהדרגה את Objective-C והפכה לשפת הפיתוח המרכזית:

```swift
// כתיבה ב-Swift
let name = "Fang"
print("Hello, \(name)")
```

עם זאת, Objective-C לא נעלמה מהעולם. היא עדיין נמצאת בשימוש נרחב בפרויקטים ותיקים ובמסגרות מערכת ברמה נמוכה. היכרות עם התחביר הבסיסי שלה עדיין יכולה להיות שימושית לתחזוקה של פרויקטים ישנים ולהבנת שכבות עמוקות יותר של המערכת.

### ההיסטוריה של SwiftUI

לפני הופעת SwiftUI, iOS השתמשה ב-**UIKit**, ו-macOS השתמשה ב-**AppKit**. שתי המסגרות הללו פעלו בגישת תכנות “אימפרטיבית”. מפתחים היו צריכים לגרור רכיבים ב-Storyboard או לכתוב קוד ידני כדי לשלוט במצב התצוגה. התוצאה הייתה הרבה קוד ועלויות תחזוקה גבוהות, במיוחד כאשר לוגיקת הממשק הייתה מורכבת.

![storyboard](../../RESOURCE/010_xcode.png)

בשנת 2019, Apple הכריזה רשמית על SwiftUI ב-WWDC. SwiftUI הביאה איתה גישת תכנות “דקלרטיבית”, ופישטה מאוד את תהליך הפיתוח של UI.

![storyboard](../../RESOURCE/010_xcode1.png)

חשוב לשים לב ש-SwiftUI אינה מימוש תחתון עצמאי לחלוטין. בפלטפורמות השונות, היא למעשה מגשרת ופועלת יחד עם UIKit (ב-iOS) או AppKit (ב-macOS), וברמה התחתונה היא עדיין נשענת על המסגרות האלו.

### הקשר בין Swift ל-UIKit/AppKit

למרות ש-Swift היא שפת תכנות כללית שיכולה לרוץ על פלטפורמות Apple השונות, היא לא יכולה להחליף לגמרי את UIKit או את AppKit. עבור צרכים מורכבים של ממשק, או עבור יכולות ש-SwiftUI עדיין לא מכסה, עדיין יש צורך להיעזר ב-UIKit או ב-AppKit.

למשל, UIKit כבר בשלה מאוד בכל מה שקשור לניהול של view controllers, אפקטים של אנימציה, זיהוי מחוות ועוד, והיא צברה הרבה מאוד ניסיון אמיתי בשטח. SwiftUI אמנם ממשיכה להשתפר, אבל יש עדיין מצבי קצה שבהם קיימות מגבלות.

לכן מפתחים רבים משלבים בתוך אותו פרויקט גם SwiftUI וגם UIKit (או AppKit), כדי ליהנות מהיתרונות של שתיהן.

מהזווית הזו אפשר להבין את SwiftUI כמעין שכבת מעטפת ברמה גבוהה יותר מעל UIKit / AppKit. תוך כדי לימוד SwiftUI, היכרות מסוימת עם המושגים הבסיסיים של UIKit ו-AppKit יכולה לעזור מאוד בקבלת החלטות טכניות נכונות יותר, במיוחד כאשר מתחזקים פרויקטים ישנים או מממשים יכולות מורכבות.

## סיכום

**Swift**: משמשת בעיקר לכתיבת לוגיקה, עיבוד נתונים, זרימות בקרה ועוד, ואינה קשורה ישירות לפריסת הממשק.

**SwiftUI**: משמשת לבניית ממשקי משתמש בצורה דקלרטיבית, וכל הקוד שקשור לתוכן התצוגה ולפריסה שייך ל-SwiftUI.

בפיתוח אמיתי, Swift ו-SwiftUI בדרך כלל פועלות יחד: Swift מטפלת בלוגיקה, ו-SwiftUI מטפלת בממשק.

מהמעבר מ-Objective-C ו-UIKit אל Swift ו-SwiftUI, אפשר לראות שסביבת הפיתוח של Apple הולכת ונעשית מודרנית ופשוטה יותר, אבל UIKit ו-AppKit עדיין נשארות חשובות מאוד בפרויקטים מסורתיים רבים.

באמצעות הבנה טובה של היחסים בין Swift ל-SwiftUI, נוכל לפתח בצורה יעילה יותר עבור iOS ו-macOS, וגם לקבל החלטות טכניות נכונות יותר כאשר מתחזקים פרויקטים ישנים.
