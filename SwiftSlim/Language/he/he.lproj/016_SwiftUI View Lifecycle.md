# מחזור החיים של תצוגת SwiftUI

בשיעור הקודם הבנו שמשתני `@State` קשורים למחזור החיים של התצוגה, וגם למדנו את השימוש הבסיסי ב-`onAppear`.

בשיעור הזה נבין טוב יותר את מחזור החיים של תצוגות SwiftUI: איך תצוגה נוצרת ומוצגת, איך משתנים בתוך התצוגה נוצרים ונבנים מחדש, ונלמד גם על `onAppear`、`onDisappear` וגם על הבנאי `init`.

בעזרת הידע הזה נוכל לבנות תמונה כוללת וברורה יותר של כל תהליך הבנייה של תצוגות SwiftUI.

## סדר ההפעלה של האפליקציה

כאשר יוצרים פרויקט iOS, Xcode מייצר כברירת מחדל שני קבצים:

```
ContentView.swift
שםהפרויקט + App.swift
```

מתוכם, `"שםהפרויקט + App.swift"` הוא קובץ הכניסה של כל האפליקציה.

למשל:

```swift
import SwiftUI

@main
struct SwiftSlimTestApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
    }
}
```

### זרימת הרצת הקוד

כאשר מריצים את האפליקציה (על סימולטור או על מכשיר פיזי), המערכת תחפש את מילת המפתח `@main`:

```swift
@main
struct SwiftSlimTestApp: App {...}
```

אחרי שקובץ הכניסה מזוהה, הקוד מתחיל לרוץ.

תחילה נכנסים אל מבנה ה-`App`, מריצים את הקוד שבתוך `body`, ואז יוצרים `WindowGroup` וטוענים לתוכו את `ContentView`.

### התפקיד של `WindowGroup`

`WindowGroup` משמש לניהול חלונות:

```swift
WindowGroup {
    ContentView()
}
```

ב-iPad וב-macOS יש תמיכה בחלונות מרובים, בעוד שב-iPhone בדרך כלל יש רק חלון אחד.

לכן, ב-iPhone, `WindowGroup` אחראי בעיקר על ניהול המסך הראשון שמוצג.

### תהליך טעינת התצוגה

לאחר שהמערכת מאתרת את `ContentView`:

```swift
import SwiftUI

struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
            Text("Hello, world!")
        }
    }
}
```

SwiftUI מריץ את הקוד שבתוך `body`, ולאחר מכן בונה את הממשק מתוך מבנה התצוגה שמוחזר (למשל `VStack`、`Text` ועוד), ומציג אותו על המסך.

לאחר שהשלבים האלה מסתיימים, אפשר לראות את `ContentView`.

![ContentView](../../Resource/016_view1.png)

חשוב לשים לב: התפקיד של `body` הוא לייצר תצוגות, ולא לאחסן תצוגות.

זה אומר שבכל פעם שהתצוגה מתרעננת, SwiftUI מחשב מחדש את `body` ויוצר מבנה תצוגה חדש.

### לוגיקת התצוגה המקדימה

התצוגה המקדימה של Xcode (`Canvas`) והרצה אמיתית הן שני מנגנונים שונים.

למשל, אם נוסיף פלט דיבוג בקובץ הכניסה:

```swift
@main
struct SwiftSlimTestApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
                .onAppear {
                    print("Go to ContentView")
                }
        }
    }
}
```

כאשר מציגים תצוגה מקדימה של `ContentView` ב-Xcode, הביצוע לא מתחיל מ-`@main`, ולכן ה-`onAppear` שבקובץ הכניסה לא יופעל.

אבל אם מריצים את האפליקציה על סימולטור או על מכשיר אמיתי, המערכת תתחיל מ-`@main`, תפעיל את כל האפליקציה לפי הזרימה המלאה, ואז כן יופעל ה-`onAppear` של קובץ הכניסה ויופיע פלט הדיבוג.

הנקודה החשובה היא ש-Xcode Preview הוא רק “רינדור חלקי” שמיועד להציג את התצוגה הנוכחית; לעומת זאת סימולטור ומכשיר פיזי מספקים “סביבת הרצה מלאה”, שבה האפליקציה כולה אכן רצה.

לכן, כאשר בודקים פונקציונליות של אפליקציה, צריך להסתמך על סימולטור או על מכשיר אמיתי; Xcode Preview לא מסוגל לספק סביבת הרצה שלמה.

## מחזור החיים של תצוגה

בשלב הנוכחי כמעט כל הקוד שלנו מרוכז בתוך `ContentView`. אבל באפליקציה אמיתית, בדרך כלל יש כמה תצוגות, והמעברים ביניהן משתנים בהתאם לתרחישים שונים.

למשל, ב“קופת החיסכון”:

![PiggyBank](../../Resource/016_view.png)

כאשר לוחצים על “הפקדה”, מופיעה התצוגה `SaveView`; וכאשר תהליך ההפקדה מסתיים או שסוגרים את החלון, `SaveView` מוסרת.

התהליך הזה הוא בעצם מחזור החיים של תצוגה: יצירה, הצגה, ולבסוף הסרה.

### סגירת האפליקציה

כאשר סוגרים את האפליקציה, כל התצוגות מוסרות, וכל המצבים הקשורים אליהן נעלמים.

לכן גם `ContentView` וגם תצוגות אחרות מוסרות מהזיכרון, וכל מצב הריצה של האפליקציה מתאפס.

## מחזור החיים של משתנים בתוך תצוגה

ב-SwiftUI, מחזור החיים של משתנים בדרך כלל קשור לתצוגה עצמה.

למשל:

```swift
struct ContentView: View {
    @State private var name = "FangJunyu"
    var body: some View {
        var num = 10
        VStack {
            Text("num:\(num)")
            Text("name:\(name)")
        }
    }
}
```

### משתני `@State`

משתנה שהוצהר עם `@State` קשור למחזור החיים של התצוגה.

```swift
@State private var name = "FangJunyu"
```

כאשר התצוגה נוצרת, גם `name` נוצר; וכאשר התצוגה מוסרת, גם `name` נהרס.

זו בדיוק הסיבה לכך שצריך להשתמש ב-`UserDefaults` או באמצעים אחרים כדי לבצע שמירה מתמשכת של נתונים.

### משתנים בתוך `body`

משתנה שמוגדר בתוך `body`:

```swift
var num = 10
```

מחזור החיים שלו קשור ישירות לתהליך הביצוע של `body`.

כאשר ה-state של SwiftUI משתנה, למשל:

```swift
@State private var name = "FangJunyu"
name = "Hu"
```

כאשר `name` משתנה, `@State` מזהה שהנתונים השתנו ומודיע ל-SwiftUI לחשב מחדש את התצוגה, ולכן `body` מחושב מחדש.

כאשר `body` מחושב מחדש, כל הקוד שבתוכו רץ מחדש, ולכן גם משתנים שהוגדרו בתוך `body` (כמו `num`) נוצרים מחדש.

זו גם הסיבה לכך שלא מומלץ להגדיר משתנים מורכבים בתוך `body`.

בכל רענון של התצוגה, המשתנים שבתוך `body` נוצרים מחדש, וזה עלול להגדיל את עלות החישוב ולפגוע בביצועים.

ב-SwiftUI, סוגים שונים של נתונים צריכים להיות מנוהלים בדרכים שונות: נתונים שצריכים לעקוב אחרי מחזור החיים של התצוגה אפשר לשמור ב-`@State`; ואילו נתונים זמניים שנועדו רק לחישוב אפשר להשאיר בתוך `body`.

## `onAppear` ו-`onDisappear`

בשיעור הקודם כבר למדנו על `onAppear`, שמופעל כאשר התצוגה מוצגת.

```swift
.onAppear {}
```

למשל:

```swift
struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
                .imageScale(.large)
                .foregroundStyle(.tint)
            Text("Hello, world!")
        }
        .padding()
        .onAppear {
            print("Show ContentView")
        }
    }
}
```

גם בתצוגה המקדימה של Xcode וגם בהרצה אמיתית אפשר לראות את פלט הדיבוג של `onAppear`.

### `onDisappear`

הצמד של `onAppear` הוא `onDisappear`:

```swift
.onDisappear {}
```

כאשר תצוגה נסגרת, `onDisappear` יופעל.

למשל:

```swift
struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
            Text("Hello, world!")
        }
        .onAppear {
            print("Show ContentView")
        }
        .onDisappear {
            print("Close ContentView")
        }
    }
}
```

כאשר התצוגה מוסרת, הקוד שבתוך `onDisappear` יתבצע.

טיפ: בשלב הנוכחי, `ContentView` היא תצוגת השורש של האפליקציה, ולכן אי אפשר לסגור או להסיר אותה ישירות. בגלל זה כרגע קשה לראות בפועל את האפקט של `onDisappear`.

רק בהמשך, כאשר נלמד מעבר בין מסכים ופתיחה של תצוגות חדשות, נוכל לראות מתי `onDisappear` אכן מופעל.

## לוגיקת היצירה וההצגה

חשוב להבין שיצירת התצוגה והצגתה הם שני שלבים שונים.

כאשר התצוגה נוצרת, הבנאי של ה-`struct` נקרא:

```swift
init() {}
```

מכיוון שתצוגות SwiftUI הן מבני `struct`:

```swift
struct ContentView: View { ... }
```

לכן, כאשר מופע של התצוגה נוצר, הבנאי `init` מתבצע. ורק כאשר התצוגה מוצגת בפועל, מופעל `onAppear`.

למשל:

```swift
struct ContentView: View {
    init() {
        print("Create ContentView")
    }
    var body: some View {
        VStack {
            Image(systemName: "globe")
            Text("Hello, world!")
        }
        .onAppear {
            print("Show ContentView")
        }
        .onDisappear {
            print("Close ContentView")
        }
    }
}
```

כאשר מציגים את התצוגה ב-Xcode Preview, אפשר לראות פלט דיבוג כזה:

```swift
Create ContentView
Show ContentView
```

כלומר, קודם `init` יוצר את התצוגה, אחר כך הקוד של `body` מחושב, ולבסוף התצוגה מוצגת ומופעל `onAppear`.

לכן חשוב להבין ששלב היצירה ושלב ההצגה של התצוגה נפרדים זה מזה.

### הבנאי `init`

`init` הוא חלק בסיסי מתחביר Swift. גם `struct` וגם `class` יכולים להגדיר אותו, והוא משמש לאתחול אובייקטים.

```swift
init() {}
```

ב-SwiftUI, כאשר תצוגה נוצרת, המערכת קוראת ל-`init` של התצוגה. אפשר להשתמש בו כדי לקבל פרמטרים או לאתחל נתונים.

אם לא כותבים `init` ידנית, Swift תייצר ל-`struct` בנאי ברירת מחדל אוטומטי.

עבור תצוגות SwiftUI כמו `ContentView`, המשמעות היא: כאשר יוצרים את התצוגה, מתבצע `init`; וכאשר התצוגה מוצגת, מתבצע `onAppear`.

לכן `init` הוא הבנאי שמופעל בזמן יצירת התצוגה, והוא ישמש אותנו בהמשך כאשר נעביר פרמטרים או נבצע אתחול של נתונים.

## סיכום

בשיעור הזה למדנו את סדר ההפעלה של האפליקציה, כלומר את זרימת הביצוע מקובץ הכניסה ועד לקובץ `ContentView`.

הבנו את מחזור החיים של תצוגות SwiftUI: כאשר תצוגה נוצרת, `init` מתבצע; כאשר היא מוצגת על המסך, `onAppear` מתבצע; וכאשר היא מוסרת או נסגרת, `onDisappear` מתבצע.

למדנו גם על מנגנון עדכון התצוגה: התצוגה מונעת על ידי state. כאשר state כמו `@State` משתנה, SwiftUI מרענן את התצוגה, מחשב מחדש את `body`, וגם המשתנים שבתוך `body` נוצרים מחדש.

מחזור החיים של משתנים עשוי להיות קשור למחזור החיים של התצוגה, בעוד שנתונים זמניים שנמצאים בתוך `body` נוצרים מחדש בכל רענון.

הבנה של מחזורי החיים האלה ושל ההתנהגות של המשתנים עוזרת לארגן את הקוד בצורה טובה יותר, ולהפוך את הלוגיקה של האפליקציה לברורה יותר.
