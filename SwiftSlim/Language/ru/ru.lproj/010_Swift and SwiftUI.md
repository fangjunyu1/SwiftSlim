# Swift и SwiftUI

При изучении разработки для платформ Apple многие учебные материалы склонны объяснять Swift и SwiftUI по отдельности. Это легко создаёт у новичков ощущение разрыва и приводит к ошибочному впечатлению, будто это две полностью независимые технологии.

На самом деле Swift и SwiftUI — это глубоко интегрированное целое: Swift — это язык программирования, который отвечает за логику и данные; SwiftUI — это фреймворк пользовательского интерфейса, построенный на Swift и отвечающий за рендеринг представлений и взаимодействие.

Этот урок поможет вам выстроить чёткую когнитивную модель: что такое Swift, что такое SwiftUI и как они взаимодействуют в коде.

## Что такое Swift?

Swift — это современный язык программирования, созданный Apple для разработки приложений под iOS, macOS, watchOS и tvOS.

![swift](../../RESOURCE/010_swift.png)

Swift — это строго типизированный статический язык. В его дизайне сделан акцент на безопасность, благодаря чему уменьшается количество типичных ошибок программирования, таких как null pointer или выход за границы массива. Также он поддерживает бесшовную интеграцию с C и Objective-C.

В реальной разработке Swift чаще отвечает за логическую часть приложения: обработку данных, сетевые запросы, операции хранения и тому подобное.

## Что такое SwiftUI?

SwiftUI — это новый фреймворк для построения интерфейсов, выпущенный Apple в 2019 году. SwiftUI использует декларативную модель программирования и позволяет разработчикам более кратко описывать структуру интерфейса и поведение взаимодействия.

![swiftui](../../RESOURCE/010_swiftui.png)

Декларативное программирование означает, что разработчику достаточно сказать системе, что нужно отобразить, а система сама будет автоматически обновлять представление при изменении данных. Больше не нужно вручную управлять состоянием интерфейса, и это заметно снижает сложность UI-разработки.

Ядро SwiftUI составляют компоненты представлений, такие как Text, Image, Button, а также контейнеры компоновки, такие как VStack, HStack и ZStack. Эти компоненты взаимодействуют через привязку данных и управление состоянием, благодаря чему интерфейс может реагировать на изменение данных и автоматически обновляться.

## Как Swift и SwiftUI работают вместе?

Роли Swift и SwiftUI можно кратко свести к следующим пунктам:

**1. Swift: обработка логики и данных**

Swift в основном используется для управления данными, хранения состояния и выполнения логики. Такой код обычно не влияет напрямую на отрисовку интерфейса, а отвечает за обработку данных и поведения.

Например, обработка данных в Swift:

```swift
let markdown = try? String(contentsOf: url)
````

Такой код включает объявление переменных, констант, функций, управляющего потока и т. д., но напрямую не влияет на отображение интерфейса.

**2. SwiftUI: декларация и рендеринг интерфейса**

SwiftUI используется для построения макета и содержимого интерфейса. Он в первую очередь отвечает за рендеринг интерфейса и взаимодействие с представлениями.

Например, создание текстового представления с помощью SwiftUI:

```swift id="m3qv2p"
Text("SwiftSlim")
```

Это код, который используется для построения и управления элементами интерфейса и напрямую связан с пользовательским UI.

### Базовый пример

Ниже приведён простой пример SwiftUI:

```swift id="1bfh5z"
// ContentView.swift

import SwiftUI

struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
                .imageScale(.large)
                .foregroundStyle(.tint)
            Text("Hello, world!")
        }
        .padding()
    }
}

#Preview {
    ContentView()
}
```

В этом примере:

**Код SwiftUI**: VStack, Image, Text и padding относятся к SwiftUI и отвечают за отображение интерфейса и компоновку.

**Код Swift**: комментарий // в верхней части файла и import SwiftUI относятся к структуре кода Swift; внизу #Preview — это макрос для предпросмотра в Xcode, он не участвует в реальном рендеринге интерфейса.

### Продвинутый пример

В реальных проектах Swift и SwiftUI часто используются совместно, чтобы каждый выполнял ту задачу, для которой подходит лучше всего:

```swift id="g1nsvu"
struct ContentView: View {
    @State private var name = "FangJunyu"

    func printName() {
        print("name: \(name)")
    }

    var body: some View {
        VStack {
            Text(name)
            Button("Print") {
                printName()
            }
        }
        .padding()
    }
}
```

В этом примере:

**Код SwiftUI**: @State — это специальная обёртка свойства SwiftUI, используемая для объявления изменяемого состояния представления; Button и Text — это компоненты представлений SwiftUI, отвечающие за отображение интерфейса и взаимодействие с пользователем.

**Код Swift**: private var name и func printName() — это код Swift, используемый для хранения данных и выполнения логики, он не влияет напрямую на рендеринг интерфейса.

Когда пользователь нажимает кнопку, SwiftUI запускает код действия внутри кнопки:

```swift id="jpt1bb"
Button("Print") {
    printName()
}
```

Здесь Button — это компонент SwiftUI, а выполняемая функция printName() — это уже код Swift, отвечающий за конкретную логику обработки.

Такое разделение позволяет Swift и SwiftUI бесшовно работать вместе: Swift отвечает за данные и логику, а SwiftUI — за отображение пользовательского интерфейса.

## Где обычно пишут код Swift и SwiftUI?

В SwiftUI интерфейс строится из представления, возвращаемого свойством body. Поэтому весь код, описывающий интерфейс, обычно пишется внутри body.

Например:

```swift id="bse2ph"
struct ContentView: View {
    var body: some View {
        Text("Hello")
    }
}
```

В этом примере Text — это компонент представления SwiftUI, поэтому он должен находиться внутри body, так как SwiftUI читает body и на его основе строит интерфейс.

Код, не связанный напрямую с интерфейсом, например переменные, функции или логика обработки данных, обычно пишется вне body. Например:

```swift id="ib7vok"
struct ContentView: View {

    // Swift: данные или логика
    var name = "Fang"

    func printName() {
        print(name)
    }

    // SwiftUI: интерфейс
    var body: some View {
        Text(name)
    }
}
```

Следует обратить внимание, что внутри body всё ещё можно использовать синтаксис Swift, например if, for и другие управляющие конструкции. Просто они используются для управления генерацией представлений SwiftUI:

```id="9qzj5w"
if isLogin {
    Text("Welcome")
}
```

Поэтому в разработке на SwiftUI это можно понимать так: код представлений, такой как Text, Image, Button и т. д., обычно пишется внутри body; код данных и логики, такой как переменные и функции, обычно пишется вне body.

## Swift-файлы

По мере углубления в обучение мы также будем сталкиваться с архитектурой MVVM, где слои ViewModel и Model обычно состоят из чистого Swift-кода и полностью отделены от слоя представлений SwiftUI.

Например, класс для управления состоянием приложения:

```swift id="v9xgzu"
class AppStorage: ObservableObject {
    static var shared = AppStorage()
    private var isLoading = false
}
```

Или структура, описывающая модель данных:

```swift id="z2ljy7"
struct SponsorStruct: Identifiable {
    var id:String
    var icon: String
}
```

Задача таких файлов — управлять данными и сохранять их, они не отображаются напрямую в представлении, поэтому полностью относятся к Swift.

## Исторический контекст

Чтобы понять отношения между Swift и SwiftUI, полезно также знать, как развивались технологии разработки Apple. По состоянию на 2026 год этот стек прошёл через несколько этапов эволюции.

### История Swift

До появления Swift основным языком программирования для платформ Apple был Objective-C, который поддерживал смешивание с C. Его синтаксис был довольно многословным и для новичков имел высокий порог входа:

```Obj-c id="udm4u4"
// Запись на Objective-C
NSString *name = @"Fang";
NSLog(@"Hello，%@", name);
```

В 2014 году Apple представила язык Swift на WWDC. Swift предложил более современный синтаксис, более высокую безопасность типов и постепенно вытеснил Objective-C как основной язык разработки:

```swift id="e5m8wa"
// Запись на Swift
let name = "Fang"
print("Hello, \(name)")
```

Однако Objective-C не исчез полностью. Он до сих пор широко используется во многих существующих проектах и низкоуровневых фреймворках. Понимание его базового синтаксиса всё ещё полезно для поддержки старых проектов и лучшего понимания внутреннего устройства системы.

### История SwiftUI

До появления SwiftUI iOS использовал **UIKit**, а macOS — **AppKit**. Эти два фреймворка применяли «императивный» стиль программирования. Разработчики управляли состоянием представлений через Storyboard с перетаскиванием компонентов или через ручное написание кода. Это приводило к большому объёму кода и высокой стоимости поддержки, особенно когда логика интерфейса становилась сложной.

![storyboard](../../RESOURCE/010_xcode.png)

В 2019 году Apple официально представила SwiftUI на WWDC. SwiftUI привнёс «декларативную» парадигму программирования и значительно упростил процесс UI-разработки.

![storyboard](../../RESOURCE/010_xcode1.png)

Важно отметить, что SwiftUI не является полностью независимой низкоуровневой реализацией. На разных платформах он по сути мостом взаимодействует с UIKit (на iOS) или AppKit (на macOS), а внизу всё ещё опирается на работу этих двух фреймворков.

### Отношения Swift с UIKit/AppKit

Хотя Swift — это универсальный язык программирования, который может использоваться на всех платформах Apple, он не может полностью заменить UIKit или AppKit. Для некоторых сложных требований к интерфейсу или тех возможностей, которые SwiftUI пока ещё не покрывает, по-прежнему приходится использовать UIKit или AppKit.

Например, UIKit уже очень зрелый в вопросах управления сложными контроллерами представлений, анимаций, распознавания жестов и так далее. Эти возможности прошли через большое количество проверок в реальных production-проектах. SwiftUI в этих областях постоянно усиливается, но в некоторых пограничных сценариях всё ещё имеет ограничения.

Поэтому многие разработчики используют в одном проекте и SwiftUI, и UIKit (или AppKit), чтобы максимально задействовать сильные стороны каждого.

С этой точки зрения SwiftUI можно понимать как высокоуровневую обёртку над UIKit / AppKit. Изучая SwiftUI, полезно также в разумной степени понимать базовые концепции UIKit и AppKit — это помогает принимать более взвешенные технические решения при поддержке старых проектов или реализации сложной функциональности.

## Итог

**Swift**: в основном используется для написания логики, обработки данных, управляющего потока и всего, что не связано напрямую с компоновкой представления.

**SwiftUI**: используется для декларативного построения пользовательского интерфейса; код содержимого и компоновки представлений относится к SwiftUI.

В реальной разработке Swift и SwiftUI обычно используются совместно: Swift отвечает за логику, SwiftUI — за интерфейс.

От Objective-C и UIKit к Swift и SwiftUI среда разработки Apple постепенно смещалась в сторону более современного и лаконичного подхода, но UIKit и AppKit по-прежнему нельзя игнорировать во многих традиционных проектах.

Понимая отношения между Swift и SwiftUI, мы можем более эффективно заниматься разработкой под iOS/macOS, а при поддержке старых проектов — принимать более разумные технические решения.
