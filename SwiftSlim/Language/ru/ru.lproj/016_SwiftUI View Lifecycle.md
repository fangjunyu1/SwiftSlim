# Жизненный цикл представления в SwiftUI

На прошлом уроке мы узнали, что переменные `@State` следуют жизненному циклу представления, а также познакомились с базовым использованием `onAppear`.

На этом уроке мы глубже разберём жизненный цикл представления в SwiftUI: как представление создаётся и отображается, как создаются и пересоздаются переменные внутри представления, а также изучим `onAppear`, `onDisappear` и конструктор `init`.

На основе этих знаний мы сформируем целостное понимание процесса построения представления в SwiftUI.

## Порядок запуска приложения

Когда вы создаёте iOS-проект, Xcode по умолчанию генерирует два файла:

```swift id="1a4k8m"
ContentView.swift
ProjectName + App.swift
```

Среди них файл `"ProjectName + App".swift` является входной точкой всего приложения.

Например:

```swift id="2f7v9q"
import SwiftUI

@main
struct SwiftSlimTestApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
    }
}
```

### Процесс выполнения кода

Когда мы запускаем приложение (в симуляторе или на реальном устройстве), система сначала ищет ключевое слово `@main`:

```swift id="3c6n2w"
@main
struct SwiftSlimTestApp: App {...}
```

После определения входного файла начинается выполнение кода.

Сначала приложение входит в структуру `App`, выполняет код внутри `body`, затем создаёт `WindowGroup` и загружает находящийся внутри него `ContentView`.

### Роль WindowGroup

`WindowGroup` используется для управления окнами:

```swift id="4d5x1r"
WindowGroup {
    ContentView()
}
```

На iPad и macOS система поддерживает несколько окон, а на iPhone обычно есть только одно окно.

Поэтому на iPhone `WindowGroup` в основном управляет первым отображаемым экраном.

### Процесс загрузки представления

После того как система находит `ContentView`:

```swift id="5e9j3t"
import SwiftUI

struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
            Text("Hello, world!")
        }
    }
}
```

SwiftUI выполняет код внутри `body`, затем на основе возвращаемой структуры представления (например, `VStack`, `Text` и т. д.) строит интерфейс и отображает его на экране.

После завершения этих шагов мы можем увидеть `ContentView` на экране.

![ContentView](../../Resource/016_view1.png)

Нужно обратить внимание: задача `body` — **генерировать представление**, а не **хранить представление**.

Это означает, что каждый раз при обновлении представления SwiftUI заново вычисляет `body` и генерирует новую структуру представления.

### Логика предварительного просмотра

Предпросмотр Xcode (`Canvas`) и реальный запуск приложения — это два разных механизма.

Например, если добавить отладочный вывод во входной файл:

```swift id="6g2p4y"
@main
struct SwiftSlimTestApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
                .onAppear {
                    print("Go to ContentView")
                }
        }
    }
}
```

При просмотре `ContentView` в Canvas Xcode выполнение не начинается с входной точки `@main`, поэтому `onAppear` во входном файле не срабатывает.

Но если приложение запустить в симуляторе или на реальном устройстве, система начнёт выполнение с `@main`, полностью пройдёт весь путь запуска приложения, вызовет `onAppear` во входной точке и выведет отладочную информацию.

Ключевой момент в том, что Xcode Preview — это только **локальный рендеринг**, который нужен лишь для отображения текущего представления; а симулятор и реальное устройство дают **полную среду выполнения**, где запускается всё приложение целиком.

Поэтому при тестировании функциональности приложения нужно ориентироваться на симулятор или реальное устройство: Xcode Preview не предоставляет полного окружения выполнения.

## Жизненный цикл представления

На текущем этапе весь наш код сосредоточен в `ContentView`. Но в реальном приложении обычно есть несколько представлений, и между ними происходит переключение в разных сценариях.

Например, в «копилке»:

![PiggyBank](../../Resource/016_view.png)

Когда пользователь нажимает «Сохранить», может показываться представление `SaveView`; когда операция завершена или окно закрывается, `SaveView` удаляется.

Этот процесс и есть жизненный цикл представления: создание представления, его показ и последующее удаление.

### Закрытие приложения

Когда приложение закрывается, все представления удаляются, а связанное с ними состояние исчезает.

Поэтому `ContentView` и другие представления удаляются из памяти, а всё рабочее состояние приложения очищается.

## Жизненный цикл переменных в представлении

В SwiftUI жизненный цикл переменных обычно связан с жизненным циклом представления.

Например:

```swift id="7h8q6u"
struct ContentView: View {
    @State private var name = "FangJunyu"
    var body: some View {
        var num = 10
        VStack {
            Text("num:\(num)")
            Text("name:\(name)")
        }
    }
}
```

### Переменная @State

Переменная, объявленная через `@State`, привязана к жизненному циклу представления.

```swift id="8i1z7v"
@State private var name = "FangJunyu"
```

Когда представление создаётся, `name` тоже создаётся; когда представление удаляется, `name` тоже уничтожается.

Именно поэтому для постоянного хранения данных нужно использовать `UserDefaults` и другие подобные способы.

### Переменные внутри body

Переменные, определённые внутри `body`:

```swift id="9j3c8x"
var num = 10
```

Их жизненный цикл связан с самим процессом выполнения `body`.

Когда состояние SwiftUI меняется, например:

```swift id="0k4d9y"
@State private var name = "FangJunyu"
name = "Hu"
```

Когда `name` изменяется, `@State` обнаруживает изменение данных и уведомляет SwiftUI о необходимости заново вычислить представление, то есть `body` пересчитывается.

Когда `body` пересчитывается, весь код внутри `body` выполняется заново, и все переменные внутри `body` (например, `num`) создаются повторно.

Вот почему не рекомендуется определять в `body` сложные переменные.

Потому что при каждом обновлении представления переменные внутри `body` будут создаваться заново, а это увеличивает стоимость вычислений и влияет на производительность.

В SwiftUI разными типами данных нужно управлять разными способами: данные, которые должны следовать жизненному циклу представления, можно хранить в `@State`; временные вычисляемые данные можно размещать внутри `body`.

## onAppear и onDisappear

На прошлом уроке мы уже изучили `onAppear`: он вызывается, когда представление появляется на экране.

```swift id="1l5e0a"
.onAppear {}
```

Например:

```swift id="2m6f1b"
struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
                .imageScale(.large)
                .foregroundStyle(.tint)
            Text("Hello, world!")
        }
        .padding()
        .onAppear {
            print("Show ContentView")
        }
    }
}
```

В Xcode Preview или при запуске можно увидеть отладочный вывод из `onAppear`.

### onDisappear

Парой к `onAppear` является `onDisappear`:

```swift id="3n7g2c"
.onDisappear {}
```

Когда представление закрывается, вызывается `onDisappear`.

Например:

```swift id="4o8h3d"
struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
            Text("Hello, world!")
        }
        .onAppear {
            print("Show ContentView")
        }
        .onDisappear {
            print("Close ContentView")
        }
    }
}
```

Когда представление удаляется, выполняется код внутри `onDisappear`.

Подсказка: на текущем этапе используемый нами `ContentView` является корневым представлением приложения, поэтому его нельзя просто напрямую закрыть или удалить. По этой причине на данном этапе нельзя наглядно увидеть эффект выполнения `onDisappear`.

Позже, когда мы начнём изучать переходы между экранами и открытие новых представлений, можно будет увидеть вызов `onDisappear`.

## Логика создания и отображения

Важно понимать, что **создание представления** и **его отображение** — это два разных этапа.

Когда представление создаётся, вызывается конструктор структуры:

```swift id="5p9i4e"
init() {}
```

Поскольку представление SwiftUI — это структура `struct`:

```swift id="6q0j5f"
struct ContentView: View { ... }
```

то при создании экземпляра представления выполняется конструктор `init`. А когда представление показывается на экране, тогда вызывается `onAppear`.

Например:

```swift id="7r1k6g"
struct ContentView: View {
    init() {
        print("Create ContentView")
    }
    var body: some View {
        VStack {
            Image(systemName: "globe")
            Text("Hello, world!")
        }
        .onAppear {
            print("Show ContentView")
        }
        .onDisappear {
            print("Close ContentView")
        }
    }
}
```

Если просматривать это представление в Xcode, можно увидеть отладочный вывод:

```swift id="8s2l7h"
Create ContentView
Show ContentView
```

Это показывает, что сначала выполняется `init` и создаётся представление, затем вычисляется код в `body`, а затем представление отображается и вызывается `onAppear`.

Поэтому важно понимать, что этапы создания и отображения представления разделены.

### Конструктор init

`init` — это базовый синтаксис Swift. Его можно определять и в `struct`, и в `class`; он используется для инициализации объекта.

```swift id="9t3m8i"
init() {}
```

В SwiftUI, когда создаётся представление, система вызывает его метод `init`, который можно использовать для приёма параметров и инициализации данных.

Если `init` не написан вручную, Swift автоматически сгенерирует для `struct` конструктор по умолчанию.

Для представлений SwiftUI, таких как `ContentView`, логика следующая: когда представление создаётся, выполняется `init`; когда представление отображается, выполняется `onAppear`.

Следовательно, `init` — это конструктор, который вызывается в момент создания представления и позже будет использоваться при передаче параметров и начальной инициализации данных.

## Итог

На этом уроке мы изучили порядок запуска приложения: от входного файла до процесса выполнения `ContentView`.

Мы поняли жизненный цикл представления SwiftUI: при создании представления выполняется `init`, при отображении на экране вызывается `onAppear`, а при удалении или закрытии представления вызывается `onDisappear`.

Мы также изучили механизм обновления представления: представление управляется состоянием, и когда `@State` или другое состояние изменяется, SwiftUI обновляет представление, пересчитывает `body`, а переменные внутри `body` создаются заново.

Жизненный цикл переменных связан с жизненным циклом представления, а временные переменные внутри `body` пересоздаются при каждом обновлении.

Понимание этого жизненного цикла и поведения переменных помогает лучше организовывать код и делает логику приложения более ясной.
