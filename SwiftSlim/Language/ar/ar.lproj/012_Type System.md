# نظام الأنواع

في الدروس السابقة، تعلّمنا المتغيرات، وأصبحنا نعرف أن المتغير يمكنه حفظ أنواع مختلفة من البيانات.

مثلاً:

```swift
var hello = "Hello, World"
```

هنا، `hello` تحفظ نصاً، ولذلك فإن نوعها هو `String`.

في هذا الفصل، سنتعلّم بشكل منهجي مفهوم النوع (`Type`)، وكذلك الإعلان الصريح عن النوع واستنتاج النوع، حتى نفهم طبيعة المتغيرات بشكل أعمق.

## ما هو النوع؟

في Swift، لكل قيمة نوع واضح. ويحدد النوع ما هي هذه القيمة، وماذا يمكنها أن تفعل.

مثلاً:

- `42` هو `Int`، أي عدد صحيح
- `"Hello"` هو `String`
- `true` هو `Bool`

هذه القيم تنتمي إلى أنواع مختلفة، ولكل نوع استخدامه الخاص.

## الأنواع الأساسية الشائعة في Swift

في مرحلة البداية، أكثر الأنواع شيوعاً هي:

- `String`: النصوص
- `Int`: الأعداد الصحيحة
- `Double`: الأعداد العشرية ذات الدقة الأعلى
- `Bool`: القيم المنطقية (`true` / `false`)

إضافة إلى ذلك، هناك أيضاً `Float`، وهو نوع عشري كذلك، لكن دقته أقل من `Double`، لذلك يُستخدم عادةً بشكل أقل. وهناك أيضاً `Character` الذي يمثل حرفاً واحداً مثل `"A"` أو `"!"`.

### التصريح بالنوع (التصريح الصريح)

في Swift، يمكننا تحديد النوع يدوياً عند تعريف المتغير:

```swift
var age: Int = 18
var price: Double = 9.9
var name: String = "FangJunyu"
var isLogin: Bool = false
```

ويتم التصريح بالنوع عبر كتابة `: النوع` بعد اسم المتغير.

أما بالنسبة للمصفوفات، فتكون الصيغة:

```swift
var scores: [Int] = [90, 85, 78]
var tags: [String] = ["Swift", "iOS"]
```

في هذا المثال، لا يمكن لمصفوفة `scores` إلا أن تحتوي على عناصر من نوع `Int`، بينما لا يمكن لمصفوفة `tags` إلا أن تحتوي على عناصر من نوع `String`.

الإعلان الصريح عن النوع يجعل نية الشيفرة أوضح، كما يساعد في بعض الحالات على تجنب أخطاء استنتاج النوع.

## استنتاج النوع

في كثير من الحالات، لسنا مضطرين إلى الإعلان عن النوع بشكل صريح:

```swift
var age = 18         // Int
var price = 9.9      // Double
var name = "Sam"     // String
var isLogin = false  // Bool
var scores = [90, 85, 78]   // [Int]
var tags = ["Swift", "iOS"] // [String]
```

وذلك لأن مترجم Swift يستطيع استنتاج النوع تلقائياً من "القيمة الابتدائية".

## التصريح الصريح واستنتاج النوع

في الدروس السابقة، لم نؤكد على التصريح الصريح بالنوع، مثل:

```swift
var age: Int = 18
```

بل فضّلنا استنتاج النوع:

```swift
var age = 18
```

في هذا المثال، الصيغتان متكافئتان، وفي كلتا الحالتين يُحدَّد نوع `age` على أنه `Int`. لكن صيغة استنتاج النوع أكثر اختصاراً.

والسبب في أننا لم نؤكد على التصريح الصريح في البداية هو أن استنتاج النوع يقلل من المعلومات الإضافية داخل الشيفرة، مما يجعل الفهم أسهل على المبتدئ.

## لماذا نحتاج إلى الأنواع؟

Swift لغة قوية في التعامل مع الأنواع (`Strongly Typed Language`).

وهذا يعني: بمجرد تحديد نوع المتغير، لا يمكن تغييره بشكل عشوائي.

مثلاً:

```swift
var age: Int = 18

age = 20       // Success
age = "20"     // Error, Cannot assign value of type 'String' to type 'Int'

var nums: [Int] = [1, 2, 3, "Hello"]  // Error
```

القيمة الأولى `20` من نوع `Int`، ولذلك يمكن إسنادها إلى `age`. أما `"20"` فهي من نوع `String`، وهذا لا يطابق `Int`، لذلك سيظهر خطأ.

أما `nums` فهي مصفوفة من النوع `[Int]`، لذلك لا يمكنها تخزين نص داخلها.

تقيّد الأنواع طريقة استخدام البيانات، وتسمح باكتشاف مشاكل عدم تطابق الأنواع أثناء الترجمة، مما يقلل من الأخطاء ويزيد من استقرار الشيفرة وقابليتها للصيانة.

## متى يجب التصريح بالنوع بشكل صريح؟

على الرغم من أن Swift تستطيع استنتاج الأنواع تلقائياً، إلا أن هناك حالات لا يستطيع فيها المترجم معرفة النوع، أو قد يستنتجه بشكل غير صحيح. عندها نحتاج إلى التصريح اليدوي.

**1. المصفوفة الفارغة**

عند إنشاء مصفوفة فارغة، يجب أن نحدد نوعها بشكل صريح:

```swift
var nums: [Int] = []
```

أما إذا لم نحدد النوع:

```swift
var nums = []   // Error, Empty collection literal requires an explicit type
```

فإن المترجم لن يستطيع معرفة نوع العناصر التي يجب أن تحتويها المصفوفة، وبالتالي سيظهر خطأ أثناء الترجمة.

**2. تجنّب الاستنتاج الخاطئ**

```swift
var value = 10   // Int
```

في هذا المثال، على الرغم من أن `10` يمكن أن تمثل أيضاً قيمة عشرية، فإن المترجم يستنتجها افتراضياً على أنها `Int` لعدم وجود فاصلة عشرية.

وإذا كنا نريد أن يكون `value` من نوع `Double`، فعلينا التصريح بذلك صراحة:

```swift
var value: Double = 10   // Double
```

وعندما لا تكون نتيجة الاستنتاج مطابقة لما نريده، فعلينا استخدام التصريح الصريح أو تعديل شكل القيمة حتى يكون النوع صحيحاً.

**3. الأنواع المخصّصة**

سنتعلم لاحقاً الأنواع المخصّصة. وعند استخدامها، نحتاج غالباً إلى التصريح الصريح:

```swift
var info: GameInfo
var users: [User] = []
```

والسبب هنا شبيه بحالة المصفوفة الفارغة: فالمترجم لا يستطيع استنتاج النوع تلقائياً من قيمة فارغة.

## تحويل الأنواع

في Swift، لا يتم تحويل الأنواع المختلفة تلقائياً. بل يجب التحويل يدوياً.

مثلاً، إذا أردنا جمع رقمين من نوعين مختلفين:

```swift
var a: Int = 10
var b: Double = 12.5

var result = a + b   // Error, Binary operator '+' cannot be applied to operands of type 'Int' and 'Double'
```

في هذه الشيفرة، `a` من نوع `Int` و`b` من نوع `Double`، ولأن النوعين مختلفان فلا يمكن إجراء العملية مباشرة بينهما.

قبل الحساب، يجب أولاً تحويل أحدهما إلى النوع نفسه:

```swift
var result = Double(a) + b
```

وهذا يعني تحويل `a` من `Int` إلى `Double`، ثم إجراء الجمع مع `b`، وبعد ذلك إسناد الناتج إلى `result`.

من المهم الانتباه إلى أن تحويل النوع لا يغيّر نوع المتغير الأصلي. فعلى سبيل المثال، `Double(a)` لا يغيّر `a` نفسها، بل ينشئ قيمة جديدة من نوع `Double` لاستخدامها في العملية.

ويمكن تحويل أنواع أخرى بطريقة مشابهة، وذلك بكتابة النوع الهدف ثم تمرير القيمة بين الأقواس:

```text
Int()、Double()、String()
```

مثال:

```swift
let score: Int = 95
let message = "Score: " + String(score)  // Int -> String

let price: Double = 9.9
let intPrice = Int(price)  // Double -> Int, 9
```

ومع ذلك، يجب الانتباه إلى أن بعض التحويلات لا تكون "كاملة". فعند تحويل `Double` إلى `Int` مثلاً، يتم قطع الجزء العشري مباشرةً، وليس تقريب القيمة.

## الأسماء المستعارة للأنواع

في Swift يمكننا أيضاً إعطاء اسم بديل لنوع موجود مسبقاً، بحيث تصبح الشيفرة أكثر وضوحاً من حيث المعنى:

```swift
typealias UserID = Int
typealias Score = Double

var id: UserID = 1001
var grade: Score = 98.5
```

هنا، `UserID` ما زالت في جوهرها `Int`، ولم يتغير النوع نفسه، وإنما أصبح الاسم أكثر دلالة.

في المشاريع الكبيرة والمعقدة، تساعد الأسماء المستعارة على تحسين قابلية القراءة.

## مثال عملي على الأنواع

الآن لنرَ مثالاً بسيطاً يوضح دور الأنواع في التطوير العملي.

### إنشاء متغير

أولاً، ننشئ متغيراً باسم `age`:

```swift
var age = 20
```

وبما أننا نريد عرضه وتعديله في SwiftUI، فإننا نستخدم `@State`:

```swift
@State private var age = 20
```

وقد ذكرنا `private` من قبل، وهي تعني أن هذا المتغير لا يمكن استخدامه إلا داخل الواجهة الحالية.

### عرض البيانات في SwiftUI

لنعرض `age` داخل SwiftUI:

```swift
struct ContentView: View {
    @State private var age = 20
    var body: some View {
        VStack(spacing: 10) {
            Text("age: \(age)")
        }
    }
}
```

الآن ستعرض الواجهة:

```text
age: 20
```

### تعديل البيانات

بعد ذلك نضيف زرين لتعديل `age`:

```swift
Button("changeAge1") {
    age = 18
}
Button("changeAge2") {
    age = "18"
}
```

### نتيجة التشغيل

عند الضغط على `changeAge1`، سيتم إسناد القيمة `18` إلى `age`:

```swift
age = 18
```

وبما أن `18` من نوع `Int`، وهو نفس نوع `age`، فإن الإسناد يتم بنجاح.

أما `changeAge2` فسيتسبب في خطأ:

```swift
age = "18"
```

ورسالة الخطأ هي:

```text
Cannot assign value of type 'String' to type 'Int'
```

وذلك لأن `"18"` من نوع `String`، بينما `age` من نوع `Int`، ولذلك لا يمكن الإسناد بين نوعين مختلفين.

### لماذا يظهر الخطأ؟

النقطة الأساسية هي هذا السطر:

```swift
@State private var age = 20   // Int
```

على الرغم من أننا لم نصرّح بالنوع صراحة، فإن المترجم استنتج من القيمة الابتدائية `20` أن `age` من نوع `Int`.

وبمجرد تحديد النوع، لا يمكن إسناد قيمة من نوع مختلف لاحقاً.

كما يجب الانتباه إلى أن زر `changeAge2` لا يفشل أثناء التشغيل فقط، بل إن الخطأ يظهر أصلاً أثناء الترجمة، ولذلك لا يمكن حتى بناء التطبيق.

وهذه واحدة من مزايا Swift بوصفها لغة قوية الأنواع: فهي تستطيع اكتشاف مشاكل الأنواع مبكراً، أثناء كتابة الشيفرة.

### الصيغة الصحيحة

إذا حذفنا الشيفرة الخاطئة:

```swift
struct ContentView: View {
    @State private var age = 20
    var body: some View {
        VStack(spacing: 10) {
            Text("age: \(age)")
            Button("changeAge1") {
                age = 18
            }
        }
    }
}
```

فعند الضغط على زر `changeAge1` ستتحدث الواجهة إلى:

```text
age: 18
```

### التصريح الصريح بالنوع (اختياري)

يمكننا أيضاً توضيح النوع صراحة عند تعريف المتغير:

```swift
@State private var age: Int = 20
```

وهذه الطريقة تجعل النوع أكثر وضوحاً، لكنها ليست ضرورية دائماً في الحالات البسيطة.

## الخلاصة

في هذا الفصل تعلّمنا نظام الأنواع في Swift ودوره في التطوير العملي.

في Swift، لكل قيمة نوع واضح. ويحدد هذا النوع معنى البيانات والعمليات التي يمكن أن تشارك فيها.

Swift لغة قوية الأنواع، ولذلك بمجرد تحديد نوع المتغير، لا يمكن إسناد نوع مختلف إليه لاحقاً. ويساعد هذا القيد على اكتشاف الأخطاء أثناء الترجمة، مما يقلل مشاكل وقت التشغيل ويرفع مستوى الأمان.

يمكن تحديد نوع المتغير تلقائياً من خلال قيمته الابتدائية، أو تحديده بشكل صريح. وفي بعض الحالات، مثل المصفوفات الفارغة، أو الاستنتاج الخاطئ، أو الأنواع المخصصة، لا يستطيع المترجم استنتاج النوع بشكل صحيح، وعندها يصبح التصريح الصريح ضرورياً.

كما أن الأنواع المختلفة لا تتحول تلقائياً فيما بينها، بل يجب استخدام `Int()` أو `Double()` أو `String()` للتحويل بشكل صريح.

ويمكن أيضاً استخدام أسماء مستعارة للأنواع، وهو أمر مفيد بشكل خاص في المشاريع الكبيرة والمعقدة.

فالنوع ليس مجرد وسم للبيانات، بل هو أيضاً آلية تقييد تضمن أن يكون استخدام البيانات آمناً وواضحاً ومنضبطاً.

## معرفة إضافية - النوع الاختياري Optional

عند تحويل الأنواع، توجد عمليات قد تفشل.

مثلاً، تحويل `String` إلى `Double`:

```swift
let input = "3.14"
let pi = Double(input)  // Double?
```

هنا، `Double(input)` لا تعيد `Double`، بل:

```text
Double?
```

وهذا هو النوع الاختياري (`Optional`).

### لماذا يُعاد نوع اختياري؟

لأن محتوى النص غير مضمون. فالقيمة `"3.14"` يمكن تحويلها إلى `Double`، بينما `"Hello"` لا يمكن تحويلها.

أي أن العملية قد تنجح وقد تفشل.

ولذلك تستخدم Swift النوع الاختياري للتعبير عن هذه الحالة غير المؤكدة.

### ما هو النوع الاختياري؟

النوع الاختياري يعني: قد توجد قيمة، وقد لا توجد.

مثلاً:

```text
Double?
```

هذا النوع يعني: إذا كانت هناك قيمة، فقد يظهر عند الطباعة بالشكل `Optional(...)`، وإذا لم توجد قيمة، فستكون `nil`.

ويجب الانتباه إلى أن `nil` لا تعني `0`، ولا تعني سلسلة نصية فارغة، بل تعني "عدم وجود قيمة".

### مثال

```swift
let input = "3.14"
let pi = Double(input)
print("pi:\(pi)")
```

الناتج:

```text
pi:Optional(3.14)
```

وهذا يعني أن النص تم تحويله بنجاح إلى `Double` بقيمة `3.14`، لكن بما أن النوع هو `Double?`، فقد تم تغليفه داخل `Optional`.

أما إذا فشل التحويل:

```swift
let input = "Hello"
let pi = Double(input)

print("pi: \(pi)")
```

فسيكون الناتج:

```text
pi: nil
```

لأن `"Hello"` لا يمكن تحويلها إلى `Double`، ولذلك تُعاد `nil`.

### ما المقصود بـ `Optional(...)`؟

عندما نطبع قيمة اختيارية مباشرة:

```swift
let input = "3.14"
let pi = Double(input)
print("pi:\(pi)")
```

فإن Swift تعرض ما يسمى "الوصف الخاص بالتصحيح":

```text
Optional(3.14)
```

وهذا ليس جزءاً من البيانات نفسها، بل هو مجرد طريقة Swift في الإشارة إلى أن القيمة الحالية من نوع اختياري وأنها تحتوي على قيمة داخلية.

### فك تغليف النوع الاختياري

في التطوير العملي، نحتاج غالباً إلى استخراج القيمة الحقيقية من النوع الاختياري. وتسمى هذه العملية "فك التغليف" (`Unwrapping`).

ومن الطرق الشائعة القيام بذلك باستخدام `??` لتوفير قيمة افتراضية:

```text
??
```

ويسمى هذا العامل: عامل دمج `nil` (`Nil-Coalescing Operator`).

مثلاً:

```swift
name ?? "FangJunyu"
```

تعني هذه الشيفرة: إذا كانت `name` تحتوي على قيمة، فأعدها؛ وإذا كانت `nil`، فاستخدم القيمة الافتراضية `"FangJunyu"`.

ومن المهم أن تكون القيمة الافتراضية من النوع نفسه.

### استخدام `??` لتوفير قيمة افتراضية

```swift
let input = "30"
let pi = Double(input)
print("pi:\(pi ?? 10)")   // 30
```

الناتج:

```text
pi: 30
```

عندما ينجح تحويل `input` إلى `Double`، تُستخدم القيمة الناتجة. وإذا فشل التحويل، تُستخدم القيمة الافتراضية بعد `??`.

أما إذا فشل التحويل:

```swift
let input = "Hello"
let pi = Double(input)

print("pi: \(pi ?? 10)")
```

فسيكون الناتج:

```text
pi: 10
```

وهنا، فشل تحويل `input` إلى `Double` فعادت `nil`، ولذلك قامت `??` بتوفير القيمة الافتراضية.

النوع الاختياري يُستخدم للتعبير عن الحالات التي قد تكون فيها هناك قيمة أو لا تكون.

وفي العمليات التي قد تفشل، مثل تحويل الأنواع، تعيد Swift نوعاً اختيارياً حتى تحافظ على أمان البرنامج.

وعندما نحتاج إلى استخدام القيمة الموجودة داخله، يمكننا الاعتماد على `??` لتقديم قيمة افتراضية، بحيث نستطيع الاستمرار في التعامل مع النتيجة حتى عند ظهور `nil`.
