# Збереження суми в скарбничці

У попередньому уроці ми реалізували просту “Скарбничку”. У цьому уроці ми вивчимо, як зробити так, щоб сума в “Скарбничці” зберігалася постійно.

У цьому уроці ми головним чином розглянемо `UserDefaults` і property wrapper `@AppStorage`, а також додатково пояснимо роль `onAppear` і спосіб використання optional-типу (`nil`).

Потрібно звернути увагу, що в цьому уроці ми продовжимо використовувати код “Скарбнички”, створений у попередньому уроці.

## Проблема постійного збереження

Зараз інтерфейс “Скарбнички” дуже простий і користуватися ним зручно.

![Piggy Bank](../../Resource/014_view.png)

Але є очевидна проблема: щоразу після оновлення подання або закриття застосунку загальна сума в “Скарбничці” знову стає `0`, а вся попередня сума зникає.

Це означає, що дані насправді не зберігаються. Іншими словами, нинішня “Скарбничка” може зберігати лише тимчасові дані.

### Чому не відбувається постійне збереження?

Це тому, що ми використовуємо змінну, оголошену через `@State`:

```swift
@State private var amount = 0
````

Змінна, оголошена через `@State`, повністю залежить за своїм життєвим циклом від подання.

Коли подання створюється, `amount` ініціалізується як `0`; коли подання знищується, `amount` також знищується.

Дані в `amount` існують лише в пам’яті й не зберігаються на пристрої.

Але ми не хочемо, щоб `amount` був прив’язаний до життєвого циклу подання, тому нам потрібно реалізувати постійне збереження даних. Інакше кажучи, зберегти дані на пристрої.

### Що таке “постійне збереження даних”

Постійне збереження даних можна розуміти як перенесення даних із “тимчасової пам’яті” у “сховище пристрою”.

Коли подання закривається або застосунок завершує роботу, дані не втрачаються.

У розробці на Swift для простого “постійного збереження” можна використовувати `UserDefaults`, а для складніших даних — `SwiftData` або `CoreData`.

У цьому уроці ми спочатку вивчимо найпростіший варіант — `UserDefaults`.

## UserDefaults

`UserDefaults` використовується для збереження легковагових даних у форматі “ключ-значення” і часто застосовується для збереження `String`, `Int`, `Double`, `Bool` та інших базових типів.

### Збереження даних

`UserDefaults` використовує метод `set` для збереження даних:

```swift id="f34i5j"
UserDefaults.standard.set("FangJunyu", forKey: "name")
UserDefaults.standard.set(28, forKey: "age")
UserDefaults.standard.set(true, forKey: "isMember")
```

Перший параметр `UserDefaults` — це дані, які потрібно зберегти. Тут передаються значення типів `String`, `Int` та інших.

Другий параметр `forKey` схожий на ім’я змінної й використовується для позначення даних, що зберігаються в `UserDefaults`.

### Читання даних

`UserDefaults` використовує відповідні методи для читання:

```swift id="7dhzoq"
let name = UserDefaults.standard.string(forKey: "name")	// FangJunyu
let age = UserDefaults.standard.integer(forKey: "age")	// 28
let isMember = UserDefaults.standard.bool(forKey: "isMember")	// true
```

Під час читання потрібно використовувати метод, що відповідає конкретному типу, інакше виникнуть помилки або будуть отримані некоректні дані.

### Що потрібно знати про UserDefaults

#### 1. Повернення optional-типу

Під час читання даних із `UserDefaults` деякі методи повертають optional-тип (`Optional`).

Наприклад:

```swift id="957o4y"
let name = UserDefaults.standard.string(forKey: "name")
```

Тип, який повертається для `name`, такий:

```swift id="d5w0ev"
String?
```

Це означає, що значення може бути, а може й не бути.

**Чому повертається optional-тип?**

Тому що в `UserDefaults` для певного `Key` не обов’язково вже існує відповідне значення.

Наприклад:

```swift id="74uodw"
let name = UserDefaults.standard.string(forKey: "City") // nil
```

Якщо раніше значення для ключа `"City"` ніколи не зберігалося, результат читання буде таким:

```swift id="8d4kc9"
nil
```

Потрібно особливо підкреслити: `nil` означає відсутність даних, а не порожній рядок.

У розділі “Система типів” ми вже стикалися з подібною ситуацією: коли перетворення типу не вдається, повертається optional-тип.

У такому випадку потрібно використати `??`, щоб надати значення за замовчуванням і виконати розпакування:

```swift id="ltjo2j"
let name = UserDefaults.standard.string(forKey: "City")	?? "Rizhao"
```

Якщо `UserDefaults` зчитує значення, використовується саме воно; якщо значення немає (`nil`), використовується значення за замовчуванням.

#### 2. Повернення не-optional-типу

Під час читання `Int`, `Double`, `Bool` із `UserDefaults` повертаються не-optional-типи.

Наприклад:

```swift id="zmjlwm"
let num = UserDefaults.standard.integer(forKey: "num")	// 0
let height = UserDefaults.standard.double(forKey: "height")	// 0.0
let isBuyCar = UserDefaults.standard.bool(forKey: "isBuyCar")	// false
```

Навіть якщо для цих `Key` ніколи не зберігалися дані, результат усе одно не буде `nil`.

Натомість повертаються значення за замовчуванням:

```swift id="cqj892"
Int → 0
Double → 0.0
Bool → false
```

#### 3. Обмеження обсягу даних

`UserDefaults` підходить лише для збереження невеликої кількості даних, наприклад конфігурації застосунку, простого стану та подібної інформації.

У реальних проєктах він не підходить для збереження великого обсягу даних або даних, які часто читаються й записуються.

## Збереження суми в “Скарбничці”

Тепер ми можемо використати `UserDefaults`, щоб реалізувати логіку збереження загальної суми “Скарбнички”.

Коли користувач натискає кнопку, введена сума додається до загальної суми, а потім загальна сума зберігається в `UserDefaults`.

```swift id="0k5mc2"
Button("Save") {
    amount += number
    number = 0
	UserDefaults.standard.set(amount, forKey: "amount")	// UserDefaults
}
.buttonStyle(.borderedProminent)
```

Тепер ми реалізували логіку запису даних.

## Пояснення логіки постійного збереження

Хоча ми вже реалізували логіку запису для постійного збереження, під час реального запуску виявиться: щоразу після оновлення подання або повторного відкриття застосунку змінна `amount` усе одно дорівнює `0`.

Це тому, що:

```swift id="3h73rw"
@State private var amount = 0
```

У SwiftUI змінна `@State` повторно ініціалізується під час створення подання.

Навіть якщо значення `amount` уже збережене в `UserDefaults`, після повторного завантаження подання `amount` усе одно буде скинуто до `0`.

Це означає, що наразі ми лише зберегли дані в `UserDefaults`, але ще не відновили їх у самому поданні.

Тому під час завантаження подання потрібно вручну зчитати дані з `UserDefaults` і повторно присвоїти їх `amount`, щоб завершити всю логіку постійного збереження.

### Логічна аналогія

Цей процес можна уявити як “шкільну дошку в класі”:

Під час уроку вчитель пише на дошці. Ці записи схожі на стан поточного подання (`@State`).

Коли урок закінчується, дошку стирають, щоб підготувати її до наступного заняття. Це відповідає знищенню подання, коли дані в `@State` також очищуються.

Щоб не втратити навчальний матеріал, учитель заздалегідь зберігає його в навчальних матеріалах. Ці матеріали схожі на дані, збережені в `UserDefaults`.

Коли починається наступний урок, дошка спочатку порожня (`@State` знову ініціалізується), і вчителю потрібно відновити попередній матеріал за навчальними матеріалами (тобто зчитати його з `UserDefaults`).

Потрібно розуміти, що дошка не відновлює попередній вміст автоматично — його потрібно заново “переписати з навчальних матеріалів” (тобто прочитати з `UserDefaults`).

## Використання onAppear для читання даних

Коли подання з’являється, потрібно вручну зчитати значення з `UserDefaults` і присвоїти його змінній `amount`, щоб справді реалізувати постійне збереження даних.

У SwiftUI для виконання ініціалізаційної логіки під час появи подання можна використовувати `onAppear`:

```swift id="c7zbc2"
.onAppear {}
```

Розмістимо код читання з `UserDefaults` усередині `onAppear`:

```swift id="lr2h0q"
.onAppear {
    amount = UserDefaults.standard.integer(forKey: "amount")
}
```

Коли подання створюється, `amount` ініціалізується як `0`; коли подання з’являється, спрацьовує `onAppear`, який читає значення з `UserDefaults` і присвоює його `amount`. Так подання зможе відобразити загальну суму, збережену минулого разу.

Тепер уся логіка постійного збереження даних для “Скарбнички” завершена.

## Повний код

```swift id="do8g9z"
struct ContentView: View {
    
    @State private var amount = 0
    @State private var number = 0
    
    var body: some View {
        VStack {
            Text("\(amount)")
                .font(.largeTitle)
                .fontWeight(.bold)
            TextField("Amount", value: $number, format: .number)
                .frame(width: 100)
                .padding(5)
                .border(Color.black, width: 1)
            Button("Save") {
                amount += number
                number = 0
                UserDefaults.standard.set(amount, forKey: "amount")    // UserDefaults
            }
            .buttonStyle(.borderedProminent)
        }
        .background {
            Image("1")
        }
        .onAppear {
            amount = UserDefaults.standard.integer(forKey: "amount")
        }
    }
}
```

## Property wrapper @AppStorage

SwiftUI надає property wrapper `@AppStorage`, який може спростити логіку збереження через `UserDefaults`.

```swift id="wc65r7"
@AppStorage("amount") private var amount = 0
```

`@AppStorage` — це обгортка над `UserDefaults`, де `"amount"` відповідає ключу (`Key`) у `UserDefaults`.

Коли подання створюється, воно зчитує дані з `UserDefaults`; коли значення змінюється, воно автоматично записується назад у `UserDefaults`.

`@AppStorage` також підтримує механізм оновлення стану SwiftUI: як і `@State`, при зміні значення він оновлює подання SwiftUI.

Інакше кажучи, `@AppStorage` може одночасно реалізувати логіку читання та збереження даних, тому більше не потрібно вручну викликати `UserDefaults` і не потрібно вручну зчитувати дані через `onAppear`.

Тому ми можемо використати `@AppStorage`, щоб замінити код із `UserDefaults`:

```swift id="6az3gm"
struct ContentView: View {
    
    @AppStorage("amount") private var amount = 0
    @State private var number = 0
    
    var body: some View {
        VStack {
            Text("\(amount)")
                .font(.largeTitle)
                .fontWeight(.bold)
            TextField("Amount", value: $number, format: .number)
                .frame(width: 100)
                .padding(5)
                .border(Color.black, width: 1)
            Button("Save") {
                amount += number
                number = 0
            }
            .buttonStyle(.borderedProminent)
        }
        .background {
            Image("1")
        }
    }
}
```

## Підсумок

У цьому уроці ми головним чином вивчили такі речі:

По-перше, `UserDefaults` використовується для збереження легковагових даних. Ми використали `UserDefaults`, щоб зберегти загальну суму на пристрої. `UserDefaults` надає методи читання й запису, а також вимагає врахування випадків з optional-типами.

По-друге, змінна `@State` використовується лише для керування тимчасовим станом у поданні, а її життєвий цикл збігається з життєвим циклом подання. Коли подання створюється знову, `@State` також повторно ініціалізується.

Під час відновлення даних із `UserDefaults` ми навчилися використовувати `onAppear`, щоб читати `UserDefaults` у момент появи подання й завершувати відновлення даних.

Нарешті, ми використали `@AppStorage`, щоб замінити `UserDefaults`, і таким чином реалізували автоматичне читання даних під час завантаження подання та автоматичний запис під час зміни значення, замінивши ручний виклик `UserDefaults` і логіку `onAppear`.

Завдяки цим знанням ми можемо реалізувати базову функцію збереження даних у застосунку, щоб дані могли зберігатися постійно.

## Вправи після уроку

Тепер “Скарбничка” вже підтримує збереження суми, але її функціональність усе ще не повна.

1. Коли користувачеві потрібно зняти суму, слід продумати відповідну логіку зняття.
2. Коли користувач хоче повністю очистити суму, також потрібно реалізувати логіку скидання.
3. Якщо ви хочете додати для скарбнички назву, потрібно також додати збереження даних типу `String`.

Можна продовжити вдосконалювати ці функції на основі наявної реалізації, щоб зробити “Скарбничку” повноціннішою.

## Додаткові знання — проблема значення за замовчуванням у полі введення

Тепер ми можемо ще трохи покращити взаємодію з полем введення, щоб зробити введення зручнішим.

![textfield](../../Resource/015_view.png)

У поточній реалізації значення за замовчуванням у полі введення — `0`. Це означає, що кожного разу перед введенням нової суми користувачеві потрібно спочатку видалити `0`, а вже потім вводити нове число. Така взаємодія не є зручною.

Краще зробити так, щоб початкове значення поля введення було порожнім, а не `0`.

Це відбувається тому, що `TextField` і `number` мають двосторонню прив’язку:

```swift id="l1v9vh"
@State private var number = 0
TextField("Amount", value: $number, format: .number)
```

Коли `number` ініціалізується як `0`, поле введення також показує `0`.

Тому нам потрібно змінити `number` на порожнє значення, тобто `nil`.

```swift id="bh0q5w"
@State private var number = nil
```

У цей момент з’явиться помилка:

```swift id="8vwfa7"
Generic parameter 'Value' could not be inferred
```

Це якраз той випадок, про який ми говорили в розділі “Система типів”: коли для змінної неможливо автоматично вивести тип, його потрібно оголосити явно.

Оскільки ми встановили `number` у `nil`, а `nil` сам по собі не містить жодної інформації про тип, компілятор не може зрозуміти, чи це `String`, `Int` чи інший тип, тому й виникає помилка.

Отже, нам потрібно явно оголосити тип для `number`:

```swift id="uqlmbg"
@State private var number:Int? = nil
```

Тут `Int?` означає optional-тип (`Optional`), тобто значення `number` може бути або `Int`, або `nil`.

Таким чином, якщо під час ініціалізації встановити `nil`, поле введення буде порожнім, а не показуватиме `0`.

### Розпакування optional-типу

Коли ми змінюємо `number` на optional-тип, початковий код обчислення починає видавати помилку:

```swift id="jlbzv6"
amount += number
```

Підказка буде такою:

```swift id="r0i5q2"
Value of optional type 'Int?' must be unwrapped to a value of type 'Int'
```

Це тому, що тепер `number` може бути `nil`. Якщо напряму брати його в обчислення, це перетворюється на таке:

```swift id="zd21ma"
amount += nil
```

Очевидно, що `nil` не можна додати до `amount`, тобто це некоректна операція.

Тому, коли optional-тип бере участь в обчисленнях, його спочатку потрібно розпакувати.

Тут можна використати `??`, щоб надати значення за замовчуванням:

```swift id="3diui4"
amount += number ?? 0
```

Коли `number` дорівнює `nil`, використовується значення за замовчуванням `0`; коли `number` має значення, використовується фактично введене число.

Так ми не лише гарантуємо правильність обчислення, а й досягаємо ефекту порожнього поля введення за замовчуванням, тому користувачеві більше не потрібно вручну видаляти `0` з поля.

## Розширений код

```swift id="hwt7mk"
struct ContentView: View {
    
    @AppStorage("amount") private var amount = 0
    @State private var number:Int? = nil
    
    var body: some View {
        VStack {
            Text("\(amount)")
                .font(.largeTitle)
                .fontWeight(.bold)
            TextField("Amount", value: $number, format: .number)
                .frame(width: 100)
                .padding(5)
                .border(Color.black, width: 1)
            Button("Save") {
                amount += number ?? 0
                number = 0
            }
            .buttonStyle(.borderedProminent)
        }
        .background {
            Image("1")
        }
    }
}
```
