# Взаємодії входу

На попередньому уроці ми створили вікно входу та за допомогою кнопок і умовної логіки реалізували просту перевірку введеного вмісту.

На цьому уроці ми продовжимо працювати на основі коду з попереднього уроку та вивчимо поле введення пароля `SecureField`, вікно попередження `Alert`, а також те, як керувати відображенням різних view на основі змінних.

Завдяки цьому уроку ви зможете краще зрозуміти базовий потік взаємодії у вікні входу. Це також базова логіка, яка часто використовується в багатьох застосунках.

## Поле введення пароля

На попередньому уроці ми використовували `TextField` для введення пароля.

Однак у справжньому інтерфейсі входу пароль зазвичай не показується безпосередньо. Натомість він відображається у вигляді крапок або з іншим прихованим ефектом, що допомагає захистити конфіденційність користувача.

У SwiftUI елемент керування, спеціально призначений для введення пароля, називається `SecureField`.

Наприклад:

```swift
SecureField("Password", text: $password)
```

Його синтаксис дуже схожий на `TextField`. Йому також потрібні текст-підказка та прив’язана змінна.

Це можна просто зрозуміти так: `SecureField` також є полем введення, але воно приховує введений вміст.

Ми можемо напряму замінити `TextField`, який спочатку використовувався для введення пароля, на `SecureField`:

```swift
SecureField("input password", text: $password)
    .frame(width: 200)
    .border(Color.gray,width: 1)
```

Коли користувач вводить пароль, вміст пароля більше не буде безпосередньо відображатися в інтерфейсі.

Ефект відображення:

![password](../../Resource/020_password.png)

Тепер ми можемо ввести ім’я користувача та пароль, а потім натиснути кнопку, щоб визначити, чи правильне введення.

### Відображення повідомлень зворотного зв’язку

Раніше ми постійно використовували `print` для виведення результатів.

Вміст `print` відображається лише в консолі. Розробники можуть бачити його під час запуску програми в Xcode, але звичайні користувачі на телефоні цих повідомлень не бачать.

Це означає, що хоча програма вже завершила перевірку, користувач усе ще не знає, чи правильний введений вміст.

У цей момент нам потрібен спосіб показувати повідомлення безпосередньо в інтерфейсі.

## Вікно попередження Alert

У SwiftUI ми можемо використовувати `Alert`, щоб показати спливне вікно повідомлення.

Базове використання:

```swift
.alert("Title", isPresented: $isPresented) {
    Button("Button", role: .cancel) {}
} message: {
    Text("Information")
}
```

Цей код виглядає трохи складнішим за попередні елементи керування, але ми можемо зрозуміти його, розібравши на частини.

`Alert` в основному містить такі частини:

- `"Title"`: заголовок вікна попередження
- `isPresented`: прив’язане логічне значення, яке використовується для керування тим, чи показувати вікно попередження
- `Button(...)`: кнопка у вікні попередження
- `message:`: вміст повідомлення, що відображається у вікні попередження

Це можна просто зрозуміти так: коли прив’язана змінна стає `true`, система показує це вікно попередження.

### Alert є модифікатором

Як і багато модифікаторів, які ми вивчали раніше, `Alert` також додається після певного view.

Наприклад, його можна прикріпити після кнопки:

```swift
Button("Sign in") {
	// ...
}
.alert() {...}
```

Його також можна прикріпити після зовнішнього `VStack`:

```swift
VStack {
    // ...
}
.alert() {...}
```

Це означає, що `Alert` не є окремим view, розміщеним незалежно в інтерфейсі. Натомість це модифікатор, прикріплений до певного view.

Це можна просто зрозуміти так: `Alert` зазвичай пишеться після кнопки або після зовнішнього layout, щоб показати повідомлення, коли виконується певна умова.

### Показ Alert

Якщо ми хочемо показати `Alert`, нам потрібна змінна для керування ним.

Наприклад:

```swift
@State private var showAlert = false
```

Ця змінна має тип `Bool`, тобто вона має лише два значення: `true` або `false`.

Потім прив’яжіть її до `isPresented` у `Alert`:

```swift
isPresented: $showAlert
```

Коли `showAlert` дорівнює `false`, вікно попередження не буде показане. Коли `showAlert` дорівнює `true`, вікно попередження з’явиться.

### Показ вікна попередження за допомогою кнопки

Тепер спочатку реалізуймо найпростіший сценарій:

Коли користувач нічого не вводить і одразу натискає кнопку `Sign in`, покажіть вікно повідомлення, яке скаже йому, що він ще не ввів жодного вмісту.

Спочатку створіть змінну для керування цим вікном повідомлення:

```swift
@State private var showEmpty = false
```

Потім додайте `Alert` після `Button`:

```swift
Button("Sign in") {
    // ...
}
.alert("Alert", isPresented: $showEmpty) {
    Button("Ok", role: .cancel) {}
} message: {
    Text("No information")
}
```

Далі виконайте перевірку в коді натискання кнопки:

```swift
Button("Sign in") {
    if user.isEmpty && password.isEmpty {
        print("Empty")
        showEmpty = true
    } 
    // ...
}
```

Якщо і `user`, і `password` порожні, встановіть `showEmpty` у `true`.

А оскільки `Alert` прив’язаний до цієї змінної, коли вона стане `true`, система автоматично покаже вікно попередження.

Ефект відображення:

![alert](../../Resource/020_alert1.png)

Таким чином, після натискання кнопки користувач може одразу побачити повідомлення, і розробнику не потрібно перевіряти вивід `print` у консолі.

### Покращення кнопки та вікон попередження

Раніше ми обробляли лише випадок, коли «нічого не було заповнено».

Нам також потрібно обробити такі ситуації:

- ім’я користувача не введено
- пароль не введено
- і ім’я користувача, і пароль введені, але дані неправильні

Тому нам також потрібно підготувати відповідні повідомлення для цих випадків.

Спочатку додайте три нові змінні:

```swift
@State private var showMissUser = false
@State private var showMissPassword = false
@State private var showWrongInfo = false
```

Потім продовжуйте додавати відповідні `Alert` після кнопки:

```swift
Button("Sign in") {
	// ...
}
.alert("Alert", isPresented: $showMissUser) {
    Button("Ok", role: .cancel) {}
} message: {
    Text("Please enter your username.")
}
.alert("Alert", isPresented: $showMissPassword) {
    Button("Ok", role: .cancel) {}
} message: {
    Text("Please enter your password.")
}
.alert("Alert", isPresented: $showWrongInfo) {
    Button("Ok", role: .cancel) {}
} message: {
    Text("Username or password verification failed.")
}
```

Далі завершіть логіку прийняття рішення всередині кнопки:

```swift
Button("Sign in") {
    if user.isEmpty && password.isEmpty {
        print("Empty")
        showEmpty = true
    } else if user.isEmpty || password.isEmpty {
        if user.isEmpty {
            print("Please enter your username.")
            showMissUser = true
        } else if password.isEmpty {
            print("Please enter your password.")
            showMissPassword = true
        }
    } else {
        if user == userID && password == passWord {
            print("Success")
        } else {
            print("Input error")
            showWrongInfo = true
        }
    }
}
```

Коли кнопка визначає, що користувач не заповнив жодної інформації, пропустив одну частину інформації або не пройшов перевірку введення, вона змінює відповідну логічну змінну на `true`, після чого з’являється відповідний `Alert`.

Тут я рекомендую вам самостійно протестувати кілька ситуацій:

- нічого не вводити
- ввести лише ім’я користувача
- ввести лише пароль
- ввести і ім’я користувача, і пароль, але з неправильним вмістом
- ввести і ім’я користувача, і пароль, і вміст правильний

Це допомагає перевірити, чи є проблеми в нашому коді. Це також дуже важливий етап «тестування» в розробці застосунків.

Якщо в певному сценарії програма поводиться неправильно, це означає, що в логіці може бути проблема і в програмі з’явився `Bug`.

Те, що розробники часто називають `Bug`, означає помилку, дефект або поведінку, яка не відповідає очікуванням.

## Завершення вікна входу

Приклади `Alert`, про які ми говорили раніше, стосувалися того, як давати користувачу зворотний зв’язок, коли «введення неправильне».

Коли користувач вводить правильні ім’я користувача та пароль, ми зазвичай не показуємо «вікно попередження». Натомість ми одразу переходимо до інтерфейсу після входу.

Це включає дуже важливу концепцію: **показ різних view залежно від значення змінної.**

Наприклад:

```swift
@State private var login = false

if login {
	Text("Personal View")
} else {
	Text("Login View")
}
```

Тут ми використовуємо тип `Bool`, щоб керувати тим, яке view показується:

- якщо `login` дорівнює `true`, це означає, що вхід уже успішний, і показується вміст після входу
- якщо `login` дорівнює `false`, це означає, що вхід не був успішним, і інтерфейс входу продовжує показуватися.

Хоча оператор `if` є умовним оператором у Swift, усередині `body` у SwiftUI він може використовуватися для керування логікою відображення view.

Це означає, що SwiftUI визначає, яке view має бути показане зараз, на основі результату умови.

Ми можемо помістити цю логіку відображення в поточний `ContentView`:

```swift
struct ContentView: View {
	@State private var login = false
	var body: some View {
    	if login {
    	    Text("Login successful")
    	} else {
			// Код початкового вікна входу
		}
	}
}
```

Тут ми додали нову змінну `login`, щоб указати, чи користувач уже успішно увійшов.

Потім ми розмістили код вікна входу, який уже написали, після оператора `else`.

Логіка відображення view:

- коли `login` дорівнює `false`, це означає, що вхід ще не був успішним, тому початкове вікно входу продовжує показуватися.
- коли `login` дорівнює `true`, це означає, що вхід успішний. У цей момент вікно входу більше не показується, а замість нього показується `Login successful`.

Це означає, що `ContentView` показуватиме різний вміст view залежно від значення `login`.

Далі давайте змінимо логіку успішного входу всередині кнопки:

```swift
Button("Sign in") {
    if user.isEmpty && password.isEmpty {
		// code
    } else if user.isEmpty || password.isEmpty {
		// code
    } else {
        if user == userID && password == passWord {
            print("Success")
            login = true
        } else {
			// code
        }
    }
}
```

Коли ім’я користувача й пароль, введені користувачем, обидва правильні, ми встановлюємо `login` у `true`.

Щойно значення `login` змінюється, змінюється і результат перевірки `if` у `ContentView`, а інтерфейс перемикається з початкового вікна входу на `Login successful`.

Ефект відображення виглядає так:

![view](../../Resource/020_view.png)

Тепер ми справді застосували ідею «показувати різні view залежно від умов» на цій поточній сторінці входу.

У реальних застосунках зазвичай усе працює саме так: різні view показуються залежно від певного значення.

Наприклад, коли користувач не ввійшов у систему, показується сторінка входу. Після успішного входу користувача показується інший інтерфейс.

Основна ідея така: **використовувати зміни змінних, щоб керувати тим, який вміст показує інтерфейс.**

## Підсумок

На цьому уроці ми вивчили три важливі речі:

1. Використовувати `SecureField` для введення паролів, щоб вміст пароля не відображався безпосередньо в інтерфейсі.
2. Використовувати `Alert`, щоб показувати повідомлення користувачу, а не лише виводити їх у консоль.
3. Використовувати змінні та умови `if` для керування відображенням різних view, що дає змогу перемикатися між інтерфейсами до входу та після входу.

Завдяки цим знанням ми тепер можемо завершити вікно входу з базовими ефектами взаємодії та краще зрозуміти ідею розробки у SwiftUI: «коли стан змінюється, інтерфейс оновлюється».

## Повний код

```swift
struct ContentView: View {
    @State private var user = ""
    @State private var password = ""
    @State private var showEmpty = false
    @State private var showMissUser = false
    @State private var showMissPassword = false
    @State private var showWrongInfo = false
    @State private var login = false
    let userID = "FangJunyu"
    let passWord = "123456"
    var body: some View {
        if login {
            Text("Login successful")
        } else {
            VStack {
                Image("019_icon")
                    .resizable()
                    .scaledToFit()
                    .frame(width: 60)
                Text("Sign in to App")
                    .font(.title)
                    .fontWeight(.bold)
                HStack {
                    Text("Username")
                        .fontWeight(.bold)
                    TextField("input user", text: $user)
                        .frame(width: 200)
                        .border(Color.gray,width: 1)
                }
                HStack {
                    Text("Password")
                        .fontWeight(.bold)
                    SecureField("input password", text: $password)
                        .frame(width: 200)
                        .border(Color.gray,width: 1)
                }
                Button("Sign in") {
                    if user.isEmpty && password.isEmpty {
                        print("Empty")
                        showEmpty = true
                    } else if user.isEmpty || password.isEmpty {
                        if user.isEmpty {
                            print("Please enter your username.")
                            showMissUser = true
                        } else if password.isEmpty {
                            print("Please enter your password.")
                            showMissPassword = true
                        }
                    } else {
                        if user == userID && password == passWord {
                            print("Success")
                            login = true
                        } else {
                            print("Input error")
                            showWrongInfo = true
                        }
                    }
                }
                .buttonStyle(.borderedProminent)
                .alert("Alert", isPresented: $showEmpty) {
                    Button("Ok", role: .cancel) {}
                } message: {
                    Text("No information")
                }
                .alert("Alert", isPresented: $showMissUser) {
                    Button("Ok", role: .cancel) {}
                } message: {
                    Text("Please enter your username.")
                }
                .alert("Alert", isPresented: $showMissPassword) {
                    Button("Ok", role: .cancel) {}
                } message: {
                    Text("Please enter your password.")
                }
                .alert("Alert", isPresented: $showWrongInfo) {
                    Button("Ok", role: .cancel) {}
                } message: {
                    Text("Username or password verification failed.")
                }
                Spacer()
            }
            .padding(.top, 50)
        }
    }
}
```

## Додаткові знання - Alert з кількома кнопками

У попередній частині цього уроку ми використовували найпростіший `Alert`, який має лише одну кнопку й головним чином використовується для повідомлення користувача.

Насправді `Alert` також може містити кілька кнопок, щоб користувач міг робити різні вибори.

Наприклад:

```swift
Button("Sign in") {
	showAlert = true
}
.alert("Alert", isPresented: $showAlert) {
    Button("Ok") {
        print("click Ok")
    }
    Button("delete", role: .destructive) {
        print("click delete")
    }
    Button("cancel", role: .cancel) {
        print("click cancel")
    }
} message: {
    Text("Delete the file?")
}
```

Після натискання кнопки з’явиться вікно повідомлення з трьома варіантами: `Ok`, `delete` і `cancel`.

Ефект відображення:

![alert](../../Resource/020_alert2.png)

Натискання різних кнопок виконає відповідний код.

### Властивість `role` у Button

Тут ви помітите, що `Button` усередині `Alert` трохи відрізняється від звичайного синтаксису кнопки, який ми вивчали раніше:

```swift
Button("delete", role: .destructive) {
    print("click delete")
}
```

Тут є додатковий параметр `role`.

У SwiftUI `role` використовується, щоб повідомити системі, який тип дії представляє ця кнопка.

Деякі поширені ролі:

- `.cancel`: дія скасування
- `.destructive`: небезпечна дія, наприклад видалення
- `nil`: звичайна дія

Різні значення `role` зазвичай також призводять до різних візуальних стилів в інтерфейсі.

Наприклад, небезпечні дії часто показуються у більш помітному стилі, щоб нагадати користувачам натискати обережно.

Крім того, `role` також надає кнопці додаткову семантичну інформацію, завдяки чому засоби читання з екрана, такі як VoiceOver на пристроях Apple, можуть легше пояснити користувачу призначення кнопки.

Тому у відповідних ситуаціях додавати `role` до кнопки — це дуже хороша звичка.
