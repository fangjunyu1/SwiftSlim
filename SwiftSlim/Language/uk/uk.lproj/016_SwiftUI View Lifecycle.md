# Життєвий цикл подання SwiftUI

У попередньому уроці ми дізналися, що змінні `@State` слідують за життєвим циклом подання, а також вивчили базове використання `onAppear`.

У цьому уроці ми глибше розберемо життєвий цикл подання SwiftUI, зрозуміємо, як подання створюється і відображається, а також як створюються і перевідтворюються змінні всередині подання. Крім того, ми вивчимо `onAppear`, `onDisappear` і конструктор `init`.

На основі цих знань ми сформуємо цілісне розуміння всього процесу побудови подання у SwiftUI.

## Порядок запуску застосунку

Коли ми створюємо iOS-проєкт, Xcode за замовчуванням генерує два файли:

```swift
ContentView.swift
НазваПроєкту + App.swift
````

Серед них файл `"НазваПроєкту + App".swift` є вхідною точкою всього застосунку.

Наприклад:

```swift
import SwiftUI

@main
struct SwiftSlimTestApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
    }
}
```

### Потік виконання коду

Коли ми запускаємо App (у симуляторі або на реальному пристрої), система шукає ключове слово `@main`:

```swift
@main
struct SwiftSlimTestApp: App {...}
```

Після підтвердження вхідного файлу починається виконання коду.

Спочатку система входить у структуру `App`, виконує код усередині `body`, потім створює `WindowGroup` і завантажує вкладений у нього `ContentView`.

### Роль WindowGroup

`WindowGroup` використовується для керування вікнами:

```swift
WindowGroup {
    ContentView()
}
```

У системах iPad / macOS підтримуються кілька вікон, тоді як на iPhone зазвичай є лише одне вікно.

Тому на iPhone `WindowGroup` головним чином відповідає за керування першим екраном, що відображається.

### Процес завантаження подання

Після того як система знаходить `ContentView`:

```swift
import SwiftUI

struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
            Text("Hello, world!")
        }
    }
}
```

SwiftUI виконує код усередині `body`, а потім на основі поверненої структури подань (наприклад, `VStack`, `Text` тощо) будує інтерфейс і відображає його на екрані.

Після завершення цих кроків ми вже можемо бачити `ContentView` у поданні.

![ContentView](../../Resource/016_view1.png)

Потрібно звернути увагу, що роль `body` полягає в генеруванні подання, а не в його зберіганні.

Це означає, що щоразу під час оновлення подання SwiftUI заново обчислює `body` і створює нову структуру подань.

### Логіка подання Preview

Попередній перегляд Xcode (`Canvas`) і реальний запуск — це два різні механізми.

Наприклад, якщо додати вхідному файлу налагоджувальний вивід:

```swift
@main
struct SwiftSlimTestApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
                .onAppear {
                    print("Go to ContentView")
                }
        }
    }
}
```

Під час попереднього перегляду `ContentView` в Xcode виконання не починається з точки входу `@main`, тому `onAppear` у вхідному файлі не спрацює.

Однак якщо запустити застосунок у симуляторі або на реальному пристрої, система почне виконання з `@main`, пройде повний шлях запуску всього App, викличе `onAppear` у точці входу й виведе налагоджувальне повідомлення.

Ключова різниця в тому, що Preview в Xcode — це лише “локальний рендеринг”, який використовується тільки для показу поточного подання; симулятор і реальний пристрій надають “повне середовище виконання”, у якому запускається весь застосунок.

Тому під час тестування функціональності App слід орієнтуватися на симулятор або реальний пристрій, оскільки Xcode Preview не може надати повне середовище виконання.

## Життєвий цикл подання

На поточному етапі весь наш код зосереджений у `ContentView`. Але в реальному застосунку App зазвичай містить багато подань, які перемикаються в різних сценаріях.

Наприклад, у “Скарбничці”:

![PiggyBank](../../Resource/016_view.png)

Коли натискається “Зберегти”, відображається подання `SaveView`; коли збереження завершено або подання закривається, `SaveView` видаляється.

Цей процес і є життєвим циклом подання: від створення подання, до його показу, а потім до видалення.

### Закриття застосунку

Коли App закривається, усі подання видаляються, а пов’язані з ними стани зникають.

Тому `ContentView` та інші подання також видаляються з пам’яті, а весь стан виконання застосунку очищується.

## Життєвий цикл змінних у поданні

У SwiftUI життєвий цикл змінних зазвичай прив’язаний до подання.

Наприклад:

```swift
struct ContentView: View {
    @State private var name = "FangJunyu"
    var body: some View {
        var num = 10
        VStack {
            Text("num:\(num)")
            Text("name:\(name)")
        }
    }
}
```

### Змінні @State

Змінні, оголошені через `@State`, прив’язані до життєвого циклу подання.

```swift
@State private var name = "FangJunyu"
```

Коли подання створюється, `name` також створюється; коли подання видаляється, `name` також знищується.

Саме тому для постійного збереження даних потрібно використовувати `UserDefaults` або подібні механізми.

### Змінні всередині body

Змінна, оголошена всередині `body`:

```swift
var num = 10
```

Її життєвий цикл прив’язаний до самого процесу виконання `body`.

Коли стан SwiftUI змінюється, наприклад:

```swift
@State private var name = "FangJunyu"
name = "Hu"
```

Коли `name` змінюється, `@State` відстежує цю зміну, повідомляє SwiftUI про оновлення, і `body` обчислюється заново.

Під час повторного обчислення `body` весь код усередині `body` виконується ще раз, а змінні всередині `body` (наприклад, `num`) створюються заново.

Саме тому не рекомендується визначати складні змінні всередині `body`.

Тому що під час кожного оновлення подання змінні всередині `body` створюватимуться заново, що збільшує обсяг обчислень і впливає на продуктивність.

У SwiftUI різні типи даних слід керувати по-різному: дані, які мають слідувати за життєвим циклом подання, можна зберігати через `@State`; тимчасові обчислювані дані можна розміщувати всередині `body`.

## onAppear і onDisappear

У попередньому уроці ми вже вивчили `onAppear`: коли подання з’являється, викликається `onAppear`.

```swift
.onAppear {}
```

Наприклад:

```swift
struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
                .imageScale(.large)
                .foregroundStyle(.tint)
            Text("Hello, world!")
        }
        .padding()
        .onAppear {
            print("Show ContentView")
        }
    }
}
```

У Preview Xcode або під час запуску можна побачити налагоджувальний вивід із `onAppear`.

### onDisappear

Відповідником `onAppear` є `onDisappear`:

```swift
.onDisappear {}
```

Коли подання закривається, викликається `onDisappear`.

Наприклад:

```swift
struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
            Text("Hello, world!")
        }
        .onAppear {
            print("Show ContentView")
        }
        .onDisappear {
            print("Close ContentView")
        }
    }
}
```

Коли подання видаляється, виконується код усередині `onDisappear`.

Підказка: на поточному етапі `ContentView`, який ми використовуємо, є кореневим поданням застосунку, тому його не можна безпосередньо закрити або видалити. Через це на цьому етапі неможливо безпосередньо побачити ефект виконання `onDisappear`.

Лише пізніше, коли ми вивчатимемо навігацію між сторінками та відкриття нових подань, можна буде побачити, як викликається `onDisappear`.

## Логіка створення й відображення

Потрібно звернути увагу, що створення подання і його відображення — це два різні етапи.

Коли подання створюється, викликається конструктор `struct`:

```swift
init() {}
```

Оскільки подання SwiftUI є структурою `struct`:

```swift
struct ContentView: View { ... }
```

тому, коли екземпляр подання створюється, виконується конструктор `init`. А коли подання відображається, уже викликається метод `onAppear`.

Наприклад:

```swift
struct ContentView: View {
    init() {
        print("Create ContentView")
    }
    var body: some View {
        VStack {
            Image(systemName: "globe")
            Text("Hello, world!")
        }
        .onAppear {
            print("Show ContentView")
        }
        .onDisappear {
            print("Close ContentView")
        }
    }
}
```

Під час попереднього перегляду подання в Xcode можна побачити налагоджувальний вивід:

```swift
Create ContentView
Show ContentView
```

Це означає, що спочатку виконується `init` для створення подання, потім обчислюється код усередині `body`, а вже після цього подання відображається й виконується `onAppear`.

Тому важливо розуміти, що етап створення подання і етап його показу розділені.

### Конструктор init

`init` — це базовий синтаксис Swift. Його можна визначати і в `struct`, і в `class`, і він використовується для ініціалізації об’єкта.

```swift
init() {}
```

У SwiftUI, коли подання створюється, система викликає метод `init` цього подання. Його можна використовувати для приймання параметрів або ініціалізації даних.

Якщо `init` не написано вручну, Swift автоматично згенерує для `struct` конструктор за замовчуванням.

Для SwiftUI-подань на кшталт `ContentView` логіка така: під час створення подання виконується `init`, а під час відображення подання виконується `onAppear`.

Тому `init` — це конструктор, який виконується під час створення подання, і надалі він знадобиться нам під час передавання параметрів або ініціалізації даних.

## Підсумок

У цьому уроці ми вивчили порядок запуску застосунку: від вхідного файлу до процесу виконання `ContentView`.

Ми розібрали життєвий цикл подання SwiftUI: під час створення подання виконується `init`, під час показу на екрані — `onAppear`, а під час видалення або закриття подання — `onDisappear`.

Ми також вивчили механізм оновлення подання: подання керується станом. Коли змінюється стан, наприклад `@State`, SwiftUI оновлює подання, повторно обчислює `body`, а змінні всередині `body` також створюються заново.

Життєвий цикл змінних прив’язаний до життєвого циклу подання, а тимчасові змінні всередині `body` створюються заново під час кожного оновлення.

Розуміння цих життєвих циклів і поведінки змінних допомагає нам краще організовувати код і робити логіку застосунку зрозумілішою.
