# Ο κύκλος ζωής των SwiftUI views

Στο προηγούμενο μάθημα μάθαμε ότι οι μεταβλητές `@State` ακολουθούν τον κύκλο ζωής της view, και είδαμε επίσης τη βασική χρήση του `onAppear`.

Σε αυτό το μάθημα θα κατανοήσουμε πιο βαθιά τον κύκλο ζωής των SwiftUI views, θα δούμε πώς δημιουργούνται και εμφανίζονται τα views, πώς δημιουργούνται και επαναδημιουργούνται οι μεταβλητές μέσα στα views, και θα μάθουμε τα `onAppear`, `onDisappear` και τον constructor `init`.

Μέσα από αυτές τις γνώσεις θα σχηματίσουμε μια πιο ολοκληρωμένη εικόνα για ολόκληρη τη διαδικασία κατασκευής ενός SwiftUI view.

## Η σειρά εκκίνησης της εφαρμογής

Όταν δημιουργούμε ένα iOS project, το Xcode δημιουργεί από προεπιλογή δύο αρχεία:

```text
ContentView.swift
ΌνομαProject + App.swift
```

Ανάμεσα σε αυτά, το αρχείο `"ΌνομαProject + App".swift` είναι το σημείο εισόδου ολόκληρης της εφαρμογής.

Για παράδειγμα:

```swift
import SwiftUI

@main
struct SwiftSlimTestApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
    }
}
```

### Ροή εκτέλεσης του κώδικα

Όταν τρέχουμε την εφαρμογή, είτε σε προσομοιωτή είτε σε πραγματική συσκευή, το σύστημα αναζητά πρώτα το keyword `@main`:

```swift
@main
struct SwiftSlimTestApp: App {...}
```

Αφού επιβεβαιώσει το entry file, ξεκινά η εκτέλεση του κώδικα.

Πρώτα μπαίνει στο struct της `App`, εκτελεί τον κώδικα μέσα στο `body`, μετά δημιουργεί το `WindowGroup` και φορτώνει το `ContentView` που βρίσκεται μέσα του.

### Ο ρόλος του WindowGroup

Το `WindowGroup` χρησιμοποιείται για τη διαχείριση παραθύρων:

```swift
WindowGroup {
    ContentView()
}
```

Στο iPad και στο macOS υποστηρίζονται πολλαπλά παράθυρα, ενώ στο iPhone συνήθως υπάρχει μόνο ένα.

Άρα στο iPhone, το `WindowGroup` διαχειρίζεται κυρίως την πρώτη οθόνη που εμφανίζεται.

### Η διαδικασία φόρτωσης ενός view

Όταν το σύστημα εντοπίσει το `ContentView`:

```swift
import SwiftUI

struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
            Text("Hello, world!")
        }
    }
}
```

το SwiftUI εκτελεί τον κώδικα μέσα στο `body`, χτίζει τη διεπαφή με βάση τη δομή views που επιστρέφεται, όπως `VStack` και `Text`, και τελικά την εμφανίζει στην οθόνη.

Αφού ολοκληρωθούν αυτά τα βήματα, μπορούμε να δούμε το `ContentView`.

![ContentView](../../Resource/016_view1.png)

Πρέπει να προσέξουμε ότι το `body` δεν έχει ως ρόλο να αποθηκεύει ένα view, αλλά να το δημιουργεί.

Αυτό σημαίνει ότι κάθε φορά που ένα view ανανεώνεται, το SwiftUI ξαναυπολογίζει το `body` και δημιουργεί μια νέα δομή views.

### Η λογική της preview view

Η προεπισκόπηση του Xcode (Canvas) και η πραγματική εκτέλεση είναι δύο διαφορετικοί μηχανισμοί.

Για παράδειγμα, αν προσθέσουμε ένα debug output στο αρχείο εισόδου:

```swift
@main
struct SwiftSlimTestApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
                .onAppear {
                    print("Go to ContentView")
                }
        }
    }
}
```

Όταν κάνουμε preview του `ContentView` μέσα στο Xcode, η εκτέλεση δεν ξεκινά από το `@main`, άρα το `onAppear` του entry file δεν ενεργοποιείται.

Αν όμως τρέξουμε την εφαρμογή σε προσομοιωτή ή πραγματική συσκευή, το σύστημα θα ξεκινήσει από το `@main`, θα εκτελέσει ολόκληρη τη ροή της εφαρμογής και θα ενεργοποιήσει και το `onAppear` στο entry point, εμφανίζοντας το αντίστοιχο debug output.

Το βασικό σημείο είναι ότι το Xcode preview είναι «μερικό rendering», χρήσιμο μόνο για να δούμε την τρέχουσα view· ενώ ο προσομοιωτής και η πραγματική συσκευή προσφέρουν «πλήρες περιβάλλον εκτέλεσης», όπου εκτελείται ολόκληρη η εφαρμογή.

Επομένως, όταν δοκιμάζουμε τη λειτουργικότητα μιας εφαρμογής, θα πρέπει να βασιζόμαστε κυρίως στον προσομοιωτή ή στην πραγματική συσκευή, γιατί το Xcode preview δεν παρέχει πλήρες runtime environment.

## Ο κύκλος ζωής ενός view

Στο τρέχον στάδιο, όλος ο κώδικάς μας βρίσκεται μέσα στο `ContentView`. Ωστόσο, σε μια πραγματική εφαρμογή, συνήθως υπάρχουν πολλά διαφορετικά views, τα οποία εμφανίζονται και αλλάζουν ανάλογα με το σενάριο.

Για παράδειγμα, στον «κουμπαρά»:

![PiggyBank](../../Resource/016_view.png)

Όταν ο χρήστης πατά «Αποθήκευση», μπορεί να εμφανιστεί ένα `SaveView`. Όταν ολοκληρωθεί η αποθήκευση ή το view κλείσει, το `SaveView` αφαιρείται.

Αυτή ακριβώς η διαδικασία είναι ο κύκλος ζωής ενός view: δημιουργία, εμφάνιση και τελικά αφαίρεση του view.

### Κλείσιμο της εφαρμογής

Όταν η εφαρμογή κλείνει, όλα τα views αφαιρούνται και μαζί τους εξαφανίζονται και οι σχετικές καταστάσεις.

Άρα τόσο το `ContentView` όσο και τα υπόλοιπα views αφαιρούνται από τη μνήμη, και όλη η runtime κατάσταση της εφαρμογής καθαρίζεται.

## Ο κύκλος ζωής των μεταβλητών μέσα σε ένα view

Στο SwiftUI, ο κύκλος ζωής των μεταβλητών είναι συνήθως δεμένος με τον κύκλο ζωής του view.

Για παράδειγμα:

```swift
struct ContentView: View {
    @State private var name = "FangJunyu"
    var body: some View {
        var num = 10
        VStack {
            Text("num:\(num)")
            Text("name:\(name)")
        }
    }
}
```

### Μεταβλητές @State

Οι μεταβλητές που δηλώνονται με `@State` είναι δεμένες με τον κύκλο ζωής της view.

```swift
@State private var name = "FangJunyu"
```

Όταν δημιουργείται η view, δημιουργείται και η `name`. Όταν αφαιρείται η view, καταστρέφεται και η `name`.

Αυτός είναι ένας από τους λόγους που χρειαζόμαστε λύσεις όπως `UserDefaults` για persistent storage.

### Μεταβλητές μέσα στο body

Μια μεταβλητή που ορίζεται μέσα στο `body`:

```swift
var num = 10
```

έχει κύκλο ζωής δεμένο με την εκτέλεση του ίδιου του `body`.

Όταν αλλάξει κάποιο state του SwiftUI, για παράδειγμα:

```swift
@State private var name = "FangJunyu"
name = "Hu"
```

όταν αλλάζει το `name`, το `@State` εντοπίζει την αλλαγή και ειδοποιεί το SwiftUI να ξαναυπολογίσει τη view, οπότε το `body` υπολογίζεται από την αρχή.

Κάθε φορά που επανυπολογίζεται το `body`, εκτελείται ξανά όλος ο κώδικας μέσα του και οι μεταβλητές που βρίσκονται εκεί, όπως η `num`, δημιουργούνται ξανά.

Αυτός είναι και ο λόγος που δεν συνιστάται να ορίζουμε πολύπλοκες μεταβλητές μέσα στο `body`.

Επειδή σε κάθε refresh της view, οι μεταβλητές του `body` ξαναδημιουργούνται, αυξάνεται το κόστος υπολογισμού και μπορεί να επηρεαστεί η απόδοση.

Στο SwiftUI, διαφορετικά είδη δεδομένων πρέπει να διαχειρίζονται με διαφορετικούς τρόπους: δεδομένα που πρέπει να ακολουθούν τον κύκλο ζωής της view μπορούν να αποθηκευτούν σε `@State`, ενώ προσωρινά υπολογιζόμενα δεδομένα μπορούν να βρίσκονται μέσα στο `body`.

## onAppear και onDisappear

Στο προηγούμενο μάθημα μάθαμε ήδη το `onAppear`, το οποίο καλείται όταν εμφανίζεται το view.

```swift
.onAppear {}
```

Για παράδειγμα:

```swift
struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
                .imageScale(.large)
                .foregroundStyle(.tint)
            Text("Hello, world!")
        }
        .padding()
        .onAppear {
            print("Show ContentView")
        }
    }
}
```

Στο Xcode preview ή κατά την εκτέλεση, μπορούμε να δούμε το debug output μέσα στο `onAppear`.

### onDisappear

Το αντίστοιχο του `onAppear` είναι το `onDisappear`:

```swift
.onDisappear {}
```

Όταν ένα view κλείνει ή αφαιρείται, καλείται το `onDisappear`.

Για παράδειγμα:

```swift
struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
            Text("Hello, world!")
        }
        .onAppear {
            print("Show ContentView")
        }
        .onDisappear {
            print("Close ContentView")
        }
    }
}
```

Όταν το view αφαιρείται, εκτελείται ο κώδικας μέσα στο `onDisappear`.

Σημείωση: Στο παρόν στάδιο, το `ContentView` είναι η root view της εφαρμογής και δεν μπορεί να κλείσει ή να αφαιρεθεί απευθείας. Επομένως, σε αυτή τη φάση δεν μπορούμε ακόμη να παρατηρήσουμε εύκολα το αποτέλεσμα του `onDisappear`.

Αργότερα, όταν μάθουμε page navigation και άνοιγμα νέων views, θα μπορέσουμε να δούμε καθαρά πότε καλείται το `onDisappear`.

## Λογική δημιουργίας και εμφάνισης

Πρέπει να προσέξουμε ότι η δημιουργία ενός view και η εμφάνισή του είναι δύο διαφορετικά στάδια.

Όταν δημιουργείται ένα view, καλείται ο constructor του `struct`:

```swift
init() {}
```

Επειδή τα SwiftUI views είναι `struct`:

```swift
struct ContentView: View { ... }
```

όταν δημιουργείται ένα instance του view, εκτελείται ο constructor `init`. Όταν το view εμφανίζεται στην οθόνη, τότε καλείται το `onAppear`.

Για παράδειγμα:

```swift
struct ContentView: View {
    init() {
        print("Create ContentView")
    }
    var body: some View {
        VStack {
            Image(systemName: "globe")
            Text("Hello, world!")
        }
        .onAppear {
            print("Show ContentView")
        }
        .onDisappear {
            print("Close ContentView")
        }
    }
}
```

Αν κάνουμε preview αυτού του view στο Xcode, μπορούμε να δούμε debug output όπως:

```swift
Create ContentView
Show ContentView
```

Αυτό δείχνει ότι πρώτα εκτελείται το `init` για να δημιουργηθεί το view, μετά υπολογίζεται το `body`, και τέλος εμφανίζεται το view και εκτελείται το `onAppear`.

Άρα, πρέπει να κατανοήσουμε ότι η φάση της δημιουργίας και η φάση της εμφάνισης είναι ξεχωριστές.

### Ο constructor init

Το `init` είναι βασική σύνταξη της Swift. Μπορεί να οριστεί τόσο σε `struct` όσο και σε `class`, και χρησιμοποιείται για την αρχικοποίηση αντικειμένων.

```swift
init() {}
```

Στο SwiftUI, όταν δημιουργείται ένα view, το σύστημα καλεί το `init` του view, το οποίο μπορεί να χρησιμοποιηθεί για λήψη παραμέτρων ή αρχικοποίηση δεδομένων.

Αν δεν γράψουμε χειροκίνητα `init`, η Swift δημιουργεί αυτόματα έναν default constructor για το `struct`.

Για SwiftUI views όπως το `ContentView`, όταν δημιουργείται το view εκτελείται το `init`, και όταν εμφανίζεται εκτελείται το `onAppear`.

Επομένως, το `init` είναι ο constructor που εκτελείται κατά τη δημιουργία του view και θα μας φανεί χρήσιμο αργότερα όταν περάσουμε παραμέτρους ή αρχικοποιούμε δεδομένα.

## Σύνοψη

Σε αυτό το μάθημα μάθαμε τη σειρά εκκίνησης μιας εφαρμογής, από το entry file μέχρι την εκτέλεση του `ContentView`.

Κατανοήσαμε τον κύκλο ζωής ενός SwiftUI view: όταν δημιουργείται ένα view εκτελείται το `init`, όταν εμφανίζεται στην οθόνη εκτελείται το `onAppear`, και όταν αφαιρείται ή κλείνει εκτελείται το `onDisappear`.

Μάθαμε επίσης τον μηχανισμό ανανέωσης των views: το view καθοδηγείται από την κατάσταση (`state`). Όταν αλλάζει κάποιο state όπως το `@State`, το SwiftUI ανανεώνει το view, ξαναϋπολογίζει το `body`, και οι μεταβλητές μέσα στο `body` ξαναδημιουργούνται.

Ο κύκλος ζωής των μεταβλητών μπορεί να ακολουθεί τον κύκλο ζωής του view, ενώ οι προσωρινές μεταβλητές του `body` επαναδημιουργούνται σε κάθε refresh.

Κατανοώντας αυτούς τους κύκλους ζωής και τη συμπεριφορά των μεταβλητών, μπορούμε να οργανώνουμε καλύτερα τον κώδικά μας και να κάνουμε τη λογική της εφαρμογής πιο καθαρή.
