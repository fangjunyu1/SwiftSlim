# Αποθήκευση του ποσού του κουμπαρά

Στο προηγούμενο μάθημα υλοποιήσαμε έναν απλό «κουμπαρά». Σε αυτό το μάθημα θα μάθουμε πώς να κάνουμε το ποσό του «κουμπαρά» να αποθηκεύεται μόνιμα.

Το μάθημα επικεντρώνεται κυρίως στα `UserDefaults` και στον property wrapper `@AppStorage`, ενώ παράλληλα θα επεκταθούμε και στο πώς λειτουργεί το `onAppear`, καθώς και στη χρήση των optional types (`nil`).

Πρέπει να προσέξουμε ότι για αυτό το μάθημα χρειάζεται να συνεχίσουμε πάνω στον κώδικα του «κουμπαρά» από το προηγούμενο μάθημα.

## Το πρόβλημα της μόνιμης αποθήκευσης

Τώρα η διεπαφή του «κουμπαρά» είναι πολύ απλή και εύχρηστη.

![Piggy Bank](../../Resource/014_view.png)

Ωστόσο υπάρχει ένα εμφανές πρόβλημα: κάθε φορά που γίνεται refresh της view ή κλείνει η εφαρμογή, το συνολικό ποσό του «κουμπαρά» επιστρέφει πάντα στο `0`, και όλο το προηγούμενο ποσό χάνεται.

Αυτό σημαίνει ότι τα δεδομένα δεν έχουν αποθηκευτεί πραγματικά. Με άλλα λόγια, ο τωρινός «κουμπαράς» μπορεί να κρατά μόνο προσωρινά δεδομένα.

### Γιατί δεν γίνεται μόνιμη αποθήκευση;

Αυτό οφείλεται στο ότι χρησιμοποιούμε μεταβλητή δηλωμένη με `@State`:

```swift
@State private var amount = 0
```

Οι μεταβλητές που δηλώνονται με `@State` εξαρτώνται πλήρως από τον κύκλο ζωής της view.

Όταν δημιουργείται η view, το `amount` αρχικοποιείται στο `0`. Όταν η view καταστρέφεται, καταστρέφεται και το `amount`.

Δηλαδή, τα δεδομένα που αποθηκεύονται στο `amount` υπάρχουν μόνο στη μνήμη και δεν έχουν αποθηκευτεί στη συσκευή.

Επειδή δεν θέλουμε το `amount` να είναι δεμένο με τον κύκλο ζωής της view, χρειάζεται να υλοποιήσουμε μόνιμη αποθήκευση. Με άλλα λόγια, να αποθηκεύσουμε τα δεδομένα στη συσκευή.

### Τι είναι η «persistent storage»

Μπορούμε να κατανοήσουμε την persistent storage ως τη διαδικασία αποθήκευσης δεδομένων από την «προσωρινή μνήμη» στον «αποθηκευτικό χώρο της συσκευής».

Έτσι, όταν κλείνει η view ή η εφαρμογή, τα δεδομένα δεν χάνονται.

Στην ανάπτυξη με Swift, για απλή persistent storage μπορούμε να χρησιμοποιήσουμε `UserDefaults`. Για πιο σύνθετα δεδομένα, συνήθως χρησιμοποιούνται `SwiftData` ή `CoreData`.

Σε αυτό το μάθημα, θα ξεκινήσουμε με το πιο απλό: `UserDefaults`.

## UserDefaults

Το `UserDefaults` χρησιμοποιείται για την αποθήκευση ελαφριών δεδομένων key-value και συχνά αξιοποιείται για βασικούς τύπους όπως `String`, `Int`, `Double`, `Bool`.

### Αποθήκευση δεδομένων

Το `UserDefaults` χρησιμοποιεί τη μέθοδο `set` για να αποθηκεύει δεδομένα:

```swift
UserDefaults.standard.set("FangJunyu", forKey: "name")
UserDefaults.standard.set(28, forKey: "age")
UserDefaults.standard.set(true, forKey: "isMember")
```

Η πρώτη παράμετρος είναι τα δεδομένα που θέλουμε να αποθηκεύσουμε, εδώ δηλαδή τιμές τύπου `String`, `Int` κτλ.

Η δεύτερη παράμετρος `forKey` λειτουργεί σαν όνομα μεταβλητής, ώστε να ταυτοποιεί τα δεδομένα μέσα στο `UserDefaults`.

### Ανάγνωση δεδομένων

Το `UserDefaults` χρησιμοποιεί αντίστοιχες μεθόδους για να διαβάσει τα δεδομένα:

```swift
let name = UserDefaults.standard.string(forKey: "name")    // FangJunyu
let age = UserDefaults.standard.integer(forKey: "age")    // 28
let isMember = UserDefaults.standard.bool(forKey: "isMember")    // true
```

Κατά την ανάγνωση, πρέπει να χρησιμοποιούμε τη σωστή μέθοδο για τον αντίστοιχο τύπο, διαφορετικά θα προκύψουν σφάλματα ή λανθασμένα δεδομένα.

### Σημεία προσοχής για το UserDefaults

#### 1. Επιστροφή optional types

Όταν χρησιμοποιούμε το `UserDefaults` για ανάγνωση δεδομένων, ορισμένες μέθοδοι επιστρέφουν optional type (`Optional`).

Για παράδειγμα:

```swift
let name = UserDefaults.standard.string(forKey: "name")
```

Ο τύπος του `name` είναι:

```text
String?
```

Αυτό σημαίνει ότι μπορεί να υπάρχει τιμή, αλλά μπορεί και να μην υπάρχει.

**Γιατί επιστρέφεται optional;**

Επειδή ένα συγκεκριμένο key μέσα στο `UserDefaults` μπορεί να μην έχει καθόλου αποθηκευμένη τιμή.

Για παράδειγμα:

```swift
let name = UserDefaults.standard.string(forKey: "City") // nil
```

Αν δεν έχει αποθηκευτεί ποτέ το key `"City"`, τότε το αποτέλεσμα θα είναι:

```text
nil
```

Πρέπει να τονιστεί ότι το `nil` σημαίνει απουσία δεδομένων και όχι κενό string.

Στο μάθημα για το «σύστημα τύπων» είδαμε παρόμοια κατάσταση: όταν μια μετατροπή τύπου αποτυγχάνει, επίσης επιστρέφεται optional.

Τότε χρειάζεται να χρησιμοποιήσουμε `??` για να δώσουμε default value και να κάνουμε unwrap:

```swift
let name = UserDefaults.standard.string(forKey: "City") ?? "Rizhao"
```

Αν το `UserDefaults` βρει τιμή, χρησιμοποιείται αυτή· αν δεν βρει (`nil`), χρησιμοποιείται η προεπιλεγμένη τιμή.

#### 2. Επιστροφή μη-optional types

Όταν το `UserDefaults` διαβάζει τύπους `Int`, `Double` και `Bool`, επιστρέφει μη-optional τιμές.

Για παράδειγμα:

```swift
let num = UserDefaults.standard.integer(forKey: "num")    // 0
let height = UserDefaults.standard.double(forKey: "height")    // 0.0
let isBuyCar = UserDefaults.standard.bool(forKey: "isBuyCar")    // false
```

Ακόμα κι αν αυτά τα keys δεν είχαν αποθηκευτεί ποτέ, η επιστρεφόμενη τιμή δεν θα είναι `nil`.

Αντίθετα, επιστρέφονται προεπιλεγμένες τιμές:

```text
Int → 0
Double → 0.0
Bool → false
```

#### 3. Περιορισμός όγκου δεδομένων

Το `UserDefaults` είναι κατάλληλο μόνο για μικρές ποσότητες δεδομένων, όπως ρυθμίσεις εφαρμογής ή απλές καταστάσεις.

Στην πράξη, δεν είναι κατάλληλο για μεγάλο όγκο δεδομένων ή για δεδομένα που γράφονται και διαβάζονται πολύ συχνά.

## Αποθήκευση του ποσού του «κουμπαρά»

Τώρα μπορούμε να χρησιμοποιήσουμε το `UserDefaults` για να υλοποιήσουμε τη λογική αποθήκευσης του συνολικού ποσού του «κουμπαρά».

Όταν ο χρήστης πατά το button, το ποσό εισαγωγής προστίθεται στο συνολικό ποσό, και στη συνέχεια το συνολικό ποσό αποθηκεύεται στο `UserDefaults`.

```swift
Button("Save") {
    amount += number
    number = 0
    UserDefaults.standard.set(amount, forKey: "amount")    // UserDefaults
}
.buttonStyle(.borderedProminent)
```

Τώρα έχουμε ολοκληρώσει τη λογική εγγραφής των δεδομένων.

## Εξήγηση της λογικής persistent storage

Παρόλο που έχουμε ολοκληρώσει τη λογική αποθήκευσης, στην πραγματική εκτέλεση θα παρατηρήσουμε ότι κάθε φορά που γίνεται refresh της view ή ανοίγουμε ξανά την εφαρμογή, η μεταβλητή `amount` εξακολουθεί να είναι `0`.

Αυτό συμβαίνει επειδή:

```swift
@State private var amount = 0
```

Στο SwiftUI, οι μεταβλητές `@State` επαναρχικοποιούνται κάθε φορά που δημιουργείται η view.

Παρόλο που η τιμή του `amount` έχει αποθηκευτεί στο `UserDefaults`, όταν η view ξαναφορτώνεται, το `amount` επιστρέφει πάλι στο `0`.

Αυτό σημαίνει ότι μέχρι τώρα απλώς αποθηκεύσαμε τα δεδομένα μέσα στο `UserDefaults`, αλλά δεν τα αποκαταστήσαμε στη view.

Άρα, όταν φορτώνεται η view, πρέπει επιπλέον να διαβάζουμε χειροκίνητα τα δεδομένα από το `UserDefaults` και να τα αναθέτουμε ξανά στο `amount`, ώστε να ολοκληρωθεί πλήρως η λογική της persistent storage.

### Αναλογία με τον πίνακα στην τάξη

Μπορούμε να κατανοήσουμε αυτή τη διαδικασία σαν έναν πίνακα μέσα σε μια σχολική αίθουσα:

Κατά τη διάρκεια του μαθήματος, ο δάσκαλος γράφει περιεχόμενο στον πίνακα. Αυτό το περιεχόμενο μοιάζει με το state της τρέχουσας view (`@State`).

Όταν τελειώνει το μάθημα, ο πίνακας σβήνεται για να ετοιμαστεί για το επόμενο μάθημα. Αυτό αντιστοιχεί στο να καταστρέφεται η view, και τα δεδομένα του `@State` να χάνονται.

Για να μην χαθεί το περιεχόμενο, ο δάσκαλος το έχει αποθηκεύσει προηγουμένως σε εκπαιδευτικό υλικό. Αυτό το υλικό αντιστοιχεί στα δεδομένα που σώζονται στο `UserDefaults`.

Την επόμενη φορά που αρχίζει το μάθημα, ο πίνακας είναι πάλι άδειος (`@State` αρχικοποίηση), και ο δάσκαλος πρέπει να ξαναγράψει το παλιό περιεχόμενο με βάση το αποθηκευμένο υλικό (`UserDefaults`).

Το σημαντικό είναι ότι ο πίνακας δεν επαναφέρει αυτόματα το παλιό περιεχόμενο· πρέπει να γίνει ξανά εγγραφή μέσω της «ανάγνωσης του υλικού», δηλαδή του `UserDefaults`.

## Ανάγνωση δεδομένων με onAppear

Όταν η view εμφανίζεται, πρέπει να διαβάζουμε χειροκίνητα την τιμή από το `UserDefaults` και να την αναθέτουμε στη μεταβλητή `amount` του `@State`, ώστε να υλοποιηθεί πραγματικά η persistent storage.

Στο SwiftUI μπορούμε να χρησιμοποιήσουμε το `onAppear` για να εκτελέσουμε αρχικοποιητική λογική όταν η view εμφανίζεται:

```swift
.onAppear {}
```

Βάζουμε μέσα στο `onAppear` τον κώδικα που διαβάζει από το `UserDefaults`:

```swift
.onAppear {
    amount = UserDefaults.standard.integer(forKey: "amount")
}
```

Όταν δημιουργείται η view, το `amount` ξεκινά από `0`. Όταν η view εμφανίζεται, ενεργοποιείται το `onAppear`, διαβάζει την τιμή από το `UserDefaults` και την αναθέτει στο `amount`. Έτσι, η view μπορεί να εμφανίσει το ποσό που είχε αποθηκευτεί την προηγούμενη φορά.

Τώρα ολοκληρώθηκε όλη η λογική persistent storage του «κουμπαρά».

## Πλήρης κώδικας

```swift
struct ContentView: View {
    
    @State private var amount = 0
    @State private var number = 0
    
    var body: some View {
        VStack {
            Text("\(amount)")
                .font(.largeTitle)
                .fontWeight(.bold)
            TextField("Amount", value: $number, format: .number)
                .frame(width: 100)
                .padding(5)
                .border(Color.black, width: 1)
            Button("Save") {
                amount += number
                number = 0
                UserDefaults.standard.set(amount, forKey: "amount")    // UserDefaults
            }
            .buttonStyle(.borderedProminent)
        }
        .background {
            Image("1")
        }
        .onAppear {
            amount = UserDefaults.standard.integer(forKey: "amount")
        }
    }
}
```

## Ο property wrapper @AppStorage

Το SwiftUI παρέχει τον property wrapper `@AppStorage`, ο οποίος απλοποιεί τη λογική αποθήκευσης με `UserDefaults`.

```swift
@AppStorage("amount") private var amount = 0
```

Το `@AppStorage` είναι στην ουσία ένα wrapper γύρω από το `UserDefaults`, και το `"amount"` αντιστοιχεί στο key του `UserDefaults`.

Όταν δημιουργείται η view, διαβάζει αυτόματα την τιμή από το `UserDefaults`. Όταν η μεταβλητή αλλάζει, γράφει αυτόματα τη νέα τιμή πίσω στο `UserDefaults`.

Υποστηρίζει επίσης τον μηχανισμό state updates του SwiftUI, και όπως το `@State`, ανανεώνει τη SwiftUI view όταν αλλάζει η τιμή.

Με άλλα λόγια, το `@AppStorage` μπορεί να υλοποιήσει μόνο του τόσο τη λογική ανάγνωσης όσο και τη λογική αποθήκευσης, χωρίς να χρειάζεται χειροκίνητη χρήση του `UserDefaults` και χωρίς να χρειάζεται `onAppear`.

Έτσι, μπορούμε να αντικαταστήσουμε τον κώδικα του `UserDefaults` με `@AppStorage`:

```swift
struct ContentView: View {
    
    @AppStorage("amount") private var amount = 0
    @State private var number = 0
    
    var body: some View {
        VStack {
            Text("\(amount)")
                .font(.largeTitle)
                .fontWeight(.bold)
            TextField("Amount", value: $number, format: .number)
                .frame(width: 100)
                .padding(5)
                .border(Color.black, width: 1)
            Button("Save") {
                amount += number
                number = 0
            }
            .buttonStyle(.borderedProminent)
        }
        .background {
            Image("1")
        }
    }
}
```

## Σύνοψη

Σε αυτό το μάθημα μάθαμε κυρίως τα εξής:

Πρώτα, ότι το `UserDefaults` χρησιμοποιείται για την αποθήκευση ελαφριών δεδομένων. Το χρησιμοποιήσαμε για να αποθηκεύσουμε το συνολικό ποσό στη συσκευή. Το `UserDefaults` παρέχει μεθόδους ανάγνωσης και εγγραφής, και χρειάζεται επίσης σωστό χειρισμό optional values.

Δεύτερον, ότι οι μεταβλητές `@State` χρησιμοποιούνται μόνο για προσωρινό state μέσα στη view και ο κύκλος ζωής τους είναι δεμένος με τη view. Όταν η view ξαναδημιουργείται, το `@State` ξαναρχικοποιείται.

Όταν αποκαθιστούμε δεδομένα από το `UserDefaults`, μάθαμε να χρησιμοποιούμε `onAppear`, ώστε να διαβάζουμε το αποθηκευμένο περιεχόμενο τη στιγμή που εμφανίζεται η view.

Τέλος, χρησιμοποιήσαμε το `@AppStorage` ως αντικατάσταση του `UserDefaults`, ώστε η view να διαβάζει αυτόματα δεδομένα όταν φορτώνεται και να γράφει αυτόματα όταν αλλάζουν οι τιμές, αντικαθιστώντας έτσι τη χειροκίνητη λογική `UserDefaults + onAppear`.

Μέσα από αυτές τις γνώσεις, μπορούμε να υλοποιήσουμε μια βασική μορφή αποθήκευσης δεδομένων για εφαρμογές, ώστε τα δεδομένα να παραμένουν αποθηκευμένα μόνιμα.

## Ασκήσεις μετά το μάθημα

Τώρα ο «κουμπαράς» υποστηρίζει αποθήκευση ποσού, αλλά οι δυνατότητές του παραμένουν ελλιπείς.

1. Αν ο χρήστης θέλει να βγάλει χρήματα, χρειάζεται να σχεδιαστεί η αντίστοιχη λογική αφαίρεσης.
2. Αν ο χρήστης θέλει να αδειάσει εντελώς το ποσό, χρειάζεται επίσης λογική reset.
3. Αν θέλουμε να δώσουμε όνομα στον κουμπαρά, πρέπει να εισαγάγουμε αποθήκευση και για δεδομένα τύπου `String`.

Μπορείς να επεκτείνεις πάνω στην υπάρχουσα βάση και να ολοκληρώσεις περισσότερο τις λειτουργίες του «κουμπαρά».

## Επέκταση - το πρόβλημα της προεπιλεγμένης τιμής στο input field

Τώρα μπορούμε να βελτιώσουμε λίγο ακόμη την αλληλεπίδραση με το input field, ώστε η εμπειρία εισαγωγής να γίνει πιο βολική.

![textfield](../../Resource/015_view.png)

Στην τρέχουσα υλοποίηση, το input field έχει προεπιλεγμένη τιμή `0`. Αυτό σημαίνει ότι κάθε φορά που ο χρήστης θέλει να εισάγει ποσό, πρώτα πρέπει να σβήσει το `0` και μετά να πληκτρολογήσει τη νέα τιμή. Αυτή η εμπειρία δεν είναι ιδιαίτερα καλή.

Θα ήταν καλύτερο το input field να ξεκινά κενό, αντί να εμφανίζει `0`.

Αυτό συμβαίνει επειδή το `TextField` και το `number` έχουν two-way binding:

```swift
@State private var number = 0
TextField("Amount", value: $number, format: .number)
```

Όταν η `number` αρχικοποιείται σε `0`, το input field εμφανίζει επίσης `0`.

Άρα χρειάζεται να αλλάξουμε το `number` σε empty value, δηλαδή `nil`.

```swift
@State private var number = nil
```

Σε αυτό το σημείο θα εμφανιστεί σφάλμα:

```text
Generic parameter 'Value' could not be inferred
```

Αυτό ακριβώς είναι η περίπτωση που αναφέραμε στο μάθημα για το «σύστημα τύπων»: όταν μια μεταβλητή δεν μπορεί να συμπεράνει αυτόματα τον τύπο της, πρέπει να δηλωθεί ρητά.

Επειδή ορίζουμε το `number` ως `nil`, και το ίδιο το `nil` δεν περιέχει πληροφορία τύπου, ο compiler δεν μπορεί να ξέρει αν αυτή η μεταβλητή είναι `String`, `Int` ή κάτι άλλο.

Άρα χρειάζεται να δηλώσουμε ρητά τον τύπο της `number`:

```swift
@State private var number:Int? = nil
```

Εδώ το `Int?` σημαίνει optional type (`Optional`), δηλαδή ότι η `number` μπορεί να έχει τιμή `Int`, αλλά μπορεί επίσης να είναι και `nil`.

Έτσι, αν την αρχικοποιήσουμε με `nil`, το input field θα εμφανίζεται κενό αντί για `0`.

### Unwrapping optional type

Όταν αλλάξουμε το `number` σε optional type, ο προηγούμενος κώδικας υπολογισμού θα εμφανίσει σφάλμα:

```swift
amount += number
```

Το μήνυμα θα είναι:

```swift
Value of optional type 'Int?' must be unwrapped to a value of type 'Int'
```

Αυτό συμβαίνει επειδή η `number` μπορεί τώρα να είναι `nil`. Αν τη χρησιμοποιήσουμε απευθείας σε υπολογισμό, στην ουσία καταλήγουμε σε κάτι σαν:

```swift
amount += nil
```

Προφανώς το `nil` δεν μπορεί να προστεθεί στο `amount`, άρα αυτή η πράξη δεν είναι έγκυρη.

Επομένως, όταν χρησιμοποιούμε optional type σε υπολογισμό, πρέπει πρώτα να το κάνουμε unwrap.

Εδώ μπορούμε να χρησιμοποιήσουμε `??` για default value:

```swift
amount += number ?? 0
```

Αν το `number` είναι `nil`, θα χρησιμοποιηθεί το default `0`. Αν το `number` έχει τιμή, θα χρησιμοποιηθεί η πραγματική τιμή που εισήχθη.

Έτσι διασφαλίζουμε και σωστό υπολογισμό και ταυτόχρονα πετυχαίνουμε το input field να ξεκινά κενό, χωρίς ο χρήστης να χρειάζεται κάθε φορά να σβήνει το `0`.

## Επεκταμένος κώδικας

```swift
struct ContentView: View {
    
    @AppStorage("amount") private var amount = 0
    @State private var number:Int? = nil
    
    var body: some View {
        VStack {
            Text("\(amount)")
                .font(.largeTitle)
                .fontWeight(.bold)
            TextField("Amount", value: $number, format: .number)
                .frame(width: 100)
                .padding(5)
                .border(Color.black, width: 1)
            Button("Save") {
                amount += number ?? 0
                number = 0
            }
            .buttonStyle(.borderedProminent)
        }
        .background {
            Image("1")
        }
    }
}
```
